Overview & Purpose

The ServiceNow Service Catalog is the platform's self-service ordering layer. It allows users to browse, request, and track services and products defined by catalog administrators. From a technical standpoint it is implemented in the glide-servicecatalog Java module (~551 source files) and is composed of two root packages: com.glideapp.servicecatalog (domain layer) and com.glide.catalog (framework/utility layer).

What the Service Catalog Provides

Catalog & Category Browsing
Users navigate hierarchical catalogs and categories to find services and products. Multiple catalogs can coexist on one instance (e.g., IT, HR, Facilities).
Item Ordering
Standard catalog items, record producers, order guides, content items, and wizard-driven items are supported item types. Each produces a defined request output.
Cart Management
Shopping cart model with multi-item ordering, named carts, quantity support, and checkout producing a formal request record chain (REQ → RITM → SCTASK).
Request Fulfillment
Flow Designer flows or legacy workflows drive RITM lifecycle through approval, fulfillment, and delivery stages to closure.
Variable Management
Rich variable system with 27+ types, variable sets, multi-row variable sets (MRVS/table variables), UI policies, and client scripts controlling form behavior.
Multi-Experience
Items render in Classic platform, Service Portal, Now Mobile, Employee Center, Workspace (Seismic/UIB), and conversationally via Virtual Agent and NowAssist AI.

Key Design Patterns

  • Valve / Chain of Responsibility — ProducerValve chain handles order submission routing
  • Singleton — CatalogUtil thread-safe facade (1028 lines)
  • Strategy — PriceGenerator for pricing, IOptionMap for variable mapping
  • Factory — Question factory creates typed variable instances from type codes
  • Observer — CatalogDBListener / SCStaleCacheListener react to DB events
  • Composite — Container variables group child variables into sections
  • Repository/Cache — Multi-layer cache (TypeSafeCacheManager) for items, categories, criteria
  • Visitor — Query visitor pattern for MRVS variable node traversal
  • Template Method — ACatalogValve base for all valve implementations
  • Builder — CatalogWizard and CompositeRecordProducer construction

Core Domain Model

CatalogRecord (Abstract Base)

All catalog domain objects extend CatalogRecord, which wraps a GlideRecord and provides lazy-loaded field access, caching hooks, and common validation utilities. This ensures consistent read patterns across Catalog, Category, CatalogItem, and Cart.

Catalog (sc_catalog)

  • Top-level container. A single instance can host multiple catalogs (IT, HR, Facilities, etc.)
  • Key fields: title, active, background_color, homepage, manager
  • Items can belong to multiple catalogs via the sc_cat_item.sc_catalogs glide_list field (denormalized array — NOT a join table)
  • Catalog objects are cached in ZZ_CATALOG_CACHE_YY

Category (sc_category)

  • Hierarchical organizer — categories support nesting via the parent field
  • Key fields: title, sc_catalog, parent, active, full_description, image, icon
  • Related lists: Catalog Items, Sub-Categories, Available For (user criteria), Not Available For
  • Cached in ZZ_CATEGORY_CACHE_YY; stale cache invalidated by SCStaleCacheListener

CartItem (sc_cart_item) and Cart (sc_cart, ~1874 lines)

  • Cart: Transient container per user/session. Named "DEFAULT" by default (constant DEFAULT_CART_NAME). Multiple named carts are supported via CartRecord constructor parameters.
  • CartItem: Line item in cart. Key fields: cart, cat_item, quantity, requested_for
  • Checkout: Cart items are converted to RITM records. Cart is cleared after successful checkout.
  • Domain conflict detection: Cart checks that all items in the cart share a compatible domain before checkout.

OrderGuide (sc_cat_item_guide, ~1507 lines)

  • Three-stage guided ordering: Describe Needs (user answers OG variables) → Choose Options (rule base evaluates which items to include) → Summary (review & submit)
  • sc_cat_item_guide_rule: Conditions that determine which child items are included based on variable answers
  • Cascade Variables: When enabled, shared variable values from the OG propagate to matching variables on each child item's RITM automatically
  • PAD sequencing: Sequence ordering for multi-step presentation (ProducerAncestorDescendant)
  • Produces one REQ with one RITM per included item

Pricing Engine

  • PriceGenerator: Strategy pattern for price computation. Supports multi-currency.
  • Display modes: always (always show price), never (hide price), nonzero (show only when > 0)
  • Price fields: price (one-time), recurring_price (subscription), recurring_frequency
  • LookupSelectBox variables can add per-choice price deltas via fLookupPrice

SQL-style queries for common diagnostics
-- Find all variables on an item (direct)
item_option_new WHERE cat_item = '<item_sys_id>' ORDER BY order

-- Find all variables via variable sets
io_set_item WHERE sc_cat_item = '<item_sys_id>'
-- then: item_option_new WHERE variable_set = '<set_sys_id>'

-- Check RITM variable pool completeness

-- If count < expected item variables: partial variable creation at checkout

-- Detect stuck Flow Designer context
sys_flow_context WHERE state IN (error, waiting)
                  AND sys_created_on > '<checkout_time>'

-- Inspect MRVS integrity
sc_multi_row_question_answer WHERE node_owner_id = '<ritm_sys_id>'
ORDER BY node_index, node_question_id.name

-- Corrupted: duplicate node_index, gaps, or null node_question_id

MRVS (Multi-Row Variable Set) — Deep Dive

MRVS is the Table Variable feature — allows collecting repeating rows of structured data (e.g., ordering multiple devices with different configurations per row).

Key Technical Facts

  • item_option_new_set.type = one_to_many designates an MRVS. Standard sets are one_to_one.
  • Row storage is sparse: only variables with non-null values have a record in sc_multi_row_question_answer. A missing row cell is simply absent, not null-valued.
  • Row index (node_index) groups multiple cells into one logical row. Gaps or duplicate indices indicate corruption.
  • The RITM Variable Editor reads MRVS rows from sc_multi_row_question_answer grouped by node_index.
  • Max row enforcement: system property glide.sc.multirow_set.rows.size + attribute max_rows on the variable set.

Rendering Contexts for MRVS

  • Classic Platform: Uses window.tableVariableCache, XML templates (question_table.xml, question_table_variable_template.xml, question_table_variable_row.xml), backed by TableVariableService.js and TableVariable.js
  • Service Portal: Template files sp_form_field.xml and sp_element_sc_multi_row.xml; widget sc-multi-row-active-row; Angular directive spScMultiRowElement.js; functions: refreshMultiRowDisplayValue(), clearAllValue(), clearValue(), createRow()
  • Workspace (Seismic): Component sn-catalog-form-multi-row-variable inside sn-catalog-form; GraphQL fragment AppCatalog_MultiRowVariableSetType; method QuestionElementType.buildMultiRowVariableSetType()

MRVS Gotchas

  • Default values on MRVS inner variables: the onLoad client script fires per new row in the row's isolated Angular scope (Service Portal) or iframe form (Classic). Scripts that reference the parent item's g_form will fail silently.
  • Validation (regex, mandatory) on MRVS inner variables: enforced by UI policies and client scripts scoped to variable_set = <mrvs_set_sys_id>. Validation on the parent item does NOT cascade to inner variables automatically.
  • Reference qualifiers on MRVS inner variables: current refers to the row context, not the parent cart item. Qualifiers that rely on parent-form data fail in MRVS rows.
  • "Get Catalog Variables" Flow Designer action: if an MRVS inner variable has a null value (user left it empty), the cell node is null, causing ITableVariableCellNode.getVariableName() NullPointerException. Always null-guard MRVS variable reads in flows.

Variable Rendering Pipeline (at item load)

  1. Read all item_option_new records for the item (ordered by order field)
  2. Read all variable sets linked via io_set_item and their inner variables
  3. Construct form layout using Container variables (type 20 = Start, 26 = Split, 21 = End)
  4. Apply UI policies from catalog_ui_policy in ascending order sequence. Later policies override earlier for same variable.
  5. Fire onLoad catalog client scripts after form construction

Default Value Resolution Order

  1. item_option_new.default_value (static or script-evaluated string) — applied server-side during form render
  2. A catalog onLoad client script that calls g_form.setValue(field, value)
  3. A catalog UI policy action that sets default_value for the variable

Higher-priority overrides win. Default values are applied during Question initialization from item_option_new — by the time OptionMap.apply() runs, user has already seen and potentially changed the default.

GlideElementVariables — Scripting Bridge

  • Virtual GlideElement — does NOT exist as a MySQL column on sc_req_item
  • Registered via var_dictionary as type GlideElementVariables
  • Backed by sc_item_option table rows + serialized blob for fast reads
  • GlideElementVariables.changes(): returns true if any variable value differs from snapshot taken at load time
  • GlideElementHierarchicalVariables: extends with container parent-child awareness using SequencedQuestionSet

InternalOptionMap vs OptionMap

Both implement IOptionMap interface for plugin-level extensibility:

  • OptionMap (default): applies user permission checks — canCreate() and canWrite() on each question
  • InternalOptionMap: calls mapper.setIgnoreQuestionPermission(true) — bypasses permission checks. Do not use without Catalog Team discussion.

Order Flow: Browse → Cart → Checkout → Fulfillment

Standard Catalog Item Order Flow

1. BROWSE
   User navigates: sc_catalog → sc_category → sc_cat_item list
   User Criteria evaluated: sc_cat_item_user_criteria_mtom (allow) and _no_mtom (deny)
   Deny wins absolutely. Results cached in USER_ITEMS_CACHE per session.

2. ITEM FORM LOAD
   Server reads item_option_new (ordered by 'order' field)
   Reads variable sets via io_set_item
   Constructs form layout (Container variables group fields)
   Applies catalog_ui_policy in ascending order
   Fires onLoad catalog_script_client scripts
   Default values applied from item_option_new.default_value

3. CART
   User fills variables and clicks Add to Cart / Order Now
   sc_cart record created (or existing DEFAULT cart used)
   sc_cart_item record created: cart + cat_item + quantity + requested_for
   Variable values stored temporarily as part of the cart item state

4. CHECKOUT
   User clicks Checkout / confirms order
   CartRecord.checkoutCart() called
   ProducerValve chain dispatched (via GlideappValveProcessor)
   Server transaction:
     a. sc_request INSERT (REQ…)
     b. For each cart item: sc_req_item INSERT (RITM…)
     c. Variable values copied: sc_item_option INSERT + sc_item_option_mtom INSERT
        (one sc_item_option per variable per RITM)
     d. MRVS rows: sc_multi_row_question_answer INSERT per cell

Valve Pipeline & Script Evaluators

Valve Pattern Overview

The Valve pattern (Chain of Responsibility) is the primary dispatch mechanism for catalog item submission. GlideappValveProcessor iterates registered valves, calling before() on each. The first valve whose before() returns true "claims" the request — the chain stops there.

Valve Hierarchy

ACatalogValve (abstract base)
├── ProducerValve          — base for all ordering (standard items + record producers)
│   └── CompositeProducerValve — extends ProducerValve for Composite Record Producers (CRP)
├── ScriptCatalogValve     — extends ACatalogValve; handles script-driven processing
└── FacadeValve            — extends ACatalogValve; handles save/preview operations (not submit)

Dispatch:
GlideappValveProcessor.before() iterates all registered valves
→ Each valve checks request action (sysverb_execute_producer, sysverb_save_facade, etc.)
→ First match wins (chain stops)
→ CompositeProducerValve.before() additionally checks isNotCRP(item) — only routes for CRP items

ProducerValve — 5-Step Pipeline (~1211 lines)

StepNameDescription
1newrecordCreates target GlideRecord for the producer's table_name. Initializes the record before any variable values are applied.
2apply templateMerges field values from any catalog template associated with the producer. Template values provide pre-populated defaults.
3merge variablesOptionMap.apply() maps submitted variable answers from HTTP request parameters to target record fields via question mapping.
4script chainSequential script evaluator pipeline (see below)
5log produced recordCreates an item_produced_record entry linking the RITM/cart to the produced target record sys_id and class name.

CompositeProducerValve

  • Handles Composite Record Producers (CRP) — multi-target records where one wizard/order produces records in multiple tables
  • Extends ProducerValve with rollback support: if any target record fails to insert, all previously committed records are rolled back
  • Uses ProducerTargetVariableMapper for edit-mode variable population

Record Producers & Target Record Mapping

Record Producer Key Properties

  • table_name (sc_cat_item_producer.table_name): The GlideRecord target table. Insert runs in user's security context (not elevated) unless the RP script explicitly elevates it.
  • script: Server-side script running BEFORE the target record insert. Has access to: producer (RP GlideRecord), current (target table GlideRecord), request (submitted values). Changes to current within this script are committed.
  • create_task: Optional — creates an RITM-equivalent task in addition to the target record.

Variable to Field Mapping

Variables on a Record Producer can be mapped to fields on the target table record. The ProducerValve's merge variables step uses ProducerTargetVariableMapper to apply this mapping.

Mapping configuration stored in item_option_new via the lookup_table and lookup_label fields or via the question's mapping configuration on the producer record.

Public Record Producer Access Requirements

For a public (unauthenticated) Record Producer submission, all three must be true:

  1. sc_cat_item.active = true and assigned to an active catalog and category
  2. User criteria: either no allow list (open), or the Guest user matches the allow list. No deny criteria matching the Guest user.
  3. Target table (sc_cat_item_producer.table_name) must have a create ACL permitting the guest/public context, OR the RP script uses gs.setSession() elevation.
!

403 errors on Record Producer submit are almost always a target table ACL issue, not a catalog issue. The catalog's authentication gate passes before the ACL on the target table is evaluated.

item_produced_record Table

After a successful submission, the item_produced_record table stores the link between the request item and the produced target record:

  • request_item: FK to sc_req_item
  • record_class: class name of the produced record (e.g., incident)
  • record_id: sys_id of the produced record

This is how the RITM's "Produced Records" related list is populated and how the "Variables Editor" on an Incident form shows the catalog variable answers.

Delivery / Execution Plan Engine

The Execution Plan (formerly "Delivery Plan" or "Service Fulfillment Steps") is a structured, sequential fulfillment model that uses reusable step configurations to drive RITM fulfillment without requiring a custom flow per item.

Components

  • sc_service_fulfillment_step_configuration: Reusable step templates (Task, Custom approval, Manager approval). Fields: dynamic title, producer, order, hide flag, service fulfillment subflow.
  • sc_service_fulfillment_step_data: Per-RITM step execution instances. One row per step per RITM. Contains step state, assigned user/group, and date fields.
  • "Service Fulfillment Steps base flow": OOB Flow Designer flow that iterates sc_service_fulfillment_step_data records for a RITM and executes each step's subflow sequentially.

Known Failure: cellNode is null NullPointerException

The most common failure in the Service Fulfillment Steps base flow occurs when the "Get Catalog Variables" action tries to extract MRVS variable data. If an MRVS inner variable has no value (user left it blank), the cell node is null, causing:

Exception from Service Fulfillment Steps base flow
Cannot invoke "com.glide.catalog.component.variables.models.table.api.ITableVariableCellNode.getVariableName()"
because "cellNode" is null

Fix pattern: Add null guards before accessing any MRVS cell value in fulfillment flow scripts:

JavaScript — null guard for MRVS in flow
// In a script step accessing MRVS output from "Get Catalog Variables" action
if (inputs.mrvs_variable != null && inputs.mrvs_variable.value != null) {
    // process the value safely
    var val = inputs.mrvs_variable.value;
}

Configurations & Feature Flags

ACL Layering — How Access is Determined

Access evaluation order (ALL must pass):

1. Table ACL (sys_security_acl, type=record, operation=read/write/create/delete)
   → applies to every row of the table

2. Field ACL (sys_security_acl, type=field)
   → applies to specific columns within the table

3. Variable-level roles (item_option_new.read_roles / write_roles / create_roles)
   → applies only to that specific variable's read/write/create

4. Catalog UI Policies (catalog_ui_policy + catalog_ui_policy_action)
   → client-side show/hide/mandatory/disabled — NOT a security control, bypassed server-side

5. Catalog Client Scripts (catalog_script_client)
   → client-side dynamic behavior — NOT a security control
!

Layers 1–3 enforce security. Layers 4–5 enforce UX behavior only. A field hidden by a UI policy (layer 4) can still be read/written via direct GlideRecord or REST API — it is NOT protected. True field-level access restriction requires a field-level ACL (layer 2).

Catalog UI Policies

Structure

  • catalog_ui_policy: Scoped to a catalog_item or variable_set (field applies_to). Has condition (script or condition builder), reverse_if_false flag, order, ui_type.
  • catalog_ui_policy_action: Child records, one per variable being controlled. Fields: variable (name), visible, mandatory, disabled (each: true/false/leave alone).

Execution Order

Policies applied in ascending order value. When two policies affect the same variable, the later (higher order) policy wins for each action type independently. A policy with reverse_if_false = true sets the opposite action value when its condition evaluates false.

Common Failure Patterns

  • Policy fires in wrong experience: Check ui_type on catalog_ui_policy. A policy with ui_type = Desktop does not run in Service Portal. Use Both for cross-experience.
  • Condition not evaluating as expected: The condition runs in the browser client context. Conditions referencing server-side data are not available — use catalog client scripts for server-round-trip logic.
  • Policy on variable set not firing: Ensure applies_to = variable_set and the correct variable set sys_id is set. A policy on the parent item does not fire for variables only defined in a variable set.

Known Issues — Core Service Catalog

Symptom Cluster: Item / Variable Visibility Inconsistent

Item visible to one user but not another

Root cause mechanism: User Criteria evaluation returns different results. Most common reasons in order of frequency:

  1. One user matches a deny criteria (sc_cat_item_user_criteria_no_mtom) — deny wins absolutely over any allow
  2. An allow list exists and the non-seeing user fails all allow criteria (deny-by-absence)
  3. The session cache has a stale evaluation — if a criteria definition or group membership changed after the session started, old result is served until session restarts
  4. Domain separation: Items in a child domain not visible to parent-domain users unless in global/parent domain

Diagnostic: Query sc_cat_item_user_criteria_mtom and sc_cat_item_user_criteria_no_mtom for the item. Evaluate each user_criteria against both users. Check the User Criteria cache at /cache_inspect.do for both users' entries.

Variable visible in Classic but hidden in SP or Workspace

Root cause: Catalog UI policies have a ui_type field. A policy with ui_type = Desktop does not execute in Service Portal or Workspace. Client scripts with ui_type mismatches cause identical divergence.

Diagnostic: Read all catalog_ui_policy and catalog_script_client records for the item. Check ui_type on each.

Symptom Cluster: HTTP 500 on Catalog REST API

/api/sn_sc/v1/servicecatalog/items/<id>/save_catalog_item returning 500:

  • Scripted validation gate in a business rule on sc_cat_item or its child tables throws an unhandled exception
  • Cross-scope access violation: scripted business rule tries to read/write a record in restricted scope
  • Database constraint violation: required column in child table lacks a value due to missing input

Diagnostic: Check syslog for errors in the same time window filtered on the user sys_id and sc_cat_item.

Symptom Cluster: Catalog Item Copying Issues

Copy sets catalog_condition incorrectly

Root cause: GlideRecord.copyTo() shallow-copies fields. If catalog_condition references a specific item sys_id or category, the copy inherits the original's values literally. The copied item's catalog_condition points to the wrong item.

Diagnostic: Read sc_cat_item.catalog_condition on both the original and the copy. If identical, the copy inherited the value as-is.

Catalog item translation overwritten after scoped app upgrade

Root cause: When a scoped app is upgraded, the upgrade process may include update set records that overwrite OOB versions of sc_cat_item name or short_description, overwriting customer customizations.

Diagnostic: Check sys_update_version for the item's sys_id around the upgrade date.

Symptom Cluster: Variable Editor Behavior on RITM

Variable Editor crashes or SOW freezes

Root cause: If one sc_item_option record has a null item_option_new reference (variable definition deleted after RITM was created), the Variable Editor's rendering loop fails when it tries to resolve the variable type/label/rendering macro.

Diagnostic: Query sc_item_option_mtom for the problematic RITM. For each linked sc_item_option, verify item_option_new is not null and points to an existing, active record. Orphaned records are the cause.

Cart details returning only first item for multi-item orders

Root cause: The cart API or REST call is being invoked with sc_cart.sys_id instead of sc_cart_item.sys_id as the lookup key, returning only the first cart item.

Symptom Cluster: Locale / Internationalization

Date fields blank when session language is non-English

Root cause: Some catalog portal widget versions hardcode date format expectations instead of using session locale settings. When session language is non-English, the widget's date parser fails.

Diagnostic: Reproduce by changing session language to the affected locale. Check browser console for JavaScript date parsing errors in the SC Catalog Item widget's client controller.

Fulfillment stage order incorrect when glide.system.locale = de.DE

Root cause: In some releases, platform numeric sort uses locale-aware string comparison instead of numeric comparison when locale uses comma as decimal separator. This causes 10 to sort before 2 as a string.

Symptom Cluster: Security — Reflected XSS in Catalog Category Search

Root cause: Jelly template com.glideapp.servicecatalog_category_search accepts URL parameter pass_search_page which is used to construct a redirect URL. If not sanitized, a JavaScript payload executes in the browser.

Known Issues — Variables

Variable Value Not Persisting to RITM (Missing Variables)

Root cause: At checkout, the platform creates one sc_item_option record per variable, then links each via sc_item_option_mtom. Intermittent failures occur when:

  1. Transaction rollback: A business rule on sc_req_item or sc_item_option throws an unhandled exception mid-loop, rolling back only the variable writes while the RITM itself was already committed in an outer transaction scope.
  2. Concurrent cart manipulation: If the user or an API modifies the cart item while checkout is in progress (race condition), the variable snapshot at checkout may be incomplete.
  3. ACL on sc_item_option create: If the fulfillment context lacks create permission on sc_item_option, the loop fails silently for restricted variables.

Diagnostic: Count sc_item_option_mtom records for the RITM → compare to item_option_new count for the item. Check syslog at the checkout timestamp for exceptions.

Variable Not Saving in Variable Editor

Root causes (in order):

  1. ACL on sc_item_option write: The editing user lacks write permission. The save endpoint returns success but the value is not committed because ACL rollback happens server-side after the response.
  2. Business rule reverting the value: A before update business rule on sc_item_option resets the value to a computed or locked state.
  3. g_form.getValue naming collision: A variable named action conflicts with the g_form API — g_form.getValue('action') returns the form's submit action name, not the variable value.

List Collector Variable Not Following Reference Qualifier

Root causes:

  1. Qualifier uses javascript: notation with current context: In the catalog item form, current refers to the cart item — not the source table of the list collector. If the qualifier expects current to be the target table record, it evaluates against the wrong record type.
  2. Qualifier references another variable's value: A static qualifier cannot react to user input. Use a catalog client script with onChange to dynamically update the qualifier.

MRVS: Multi-Row Variable Set Failure Modes

MRVS fields not displaying data on RITM form

Root cause: The Variable Editor reads MRVS rows grouped by node_index. If records are corrupted (duplicate node_index, null node_question_id, inconsistent variable_set references), the MRVS renderer shows nothing or throws a rendering exception.

Common causes of corruption:

  • Failed partial save committed some row cells but not others
  • Manual deletion of item_option_new records after the RITM was created (leaves orphan cells with null io reference)
  • Interrupted mid-checkout that wrote some rows but not all

Diagnostic: Query sc_multi_row_question_answer for node_owner_id = <ritm_sys_id> and inspect for: null node_question_id (orphaned cell), duplicate (node_index, node_question_id) combinations, node_variable_set_id values that don't match any io_set_item for the item.

MRVS date variable not populating on submitted RITM

Root cause: Date variables in MRVS store their value as a string in sc_multi_row_question_answer.node_value. If the user's session locale uses a different date format than the variable's internal format expectation, the serialization may produce a value that the Variable Editor cannot parse on display, showing an empty date field.

MRVS UI Policy not firing on inner variables

Root cause: Catalog UI policies controlling inner MRVS variables must be scoped to the variable set (applies_to = variable_set), not to the parent catalog item. A UI policy on the catalog item does not propagate into the MRVS row form context. Catalog client scripts controlling MRVS inner variables must similarly target the variable_set field, not the cat_item field.

Reference Variable Display Value Issues

List Collector showing sys_ids instead of display values

Root cause: When stored sys_ids are displayed, the Variable Editor dot-walks through the reference table. Sys_ids appear when:

  1. Referenced records were deleted after submission
  2. User lacks read access to the reference table records — ACL blocks the dot-walk
  3. List contains stale or invalid sys_ids from a data migration

Known Issues — Catalog Builder

Catalog Item Stuck in Publishing State

The item remains in "Publishing" state when any of the following occur:

  • A. Validation gate failure: A mandatory field in the template is empty, or a scripted validation condition returns false. The background job halts at validation without transitioning to Published. The item stays in Publishing with no visible error in the UI — check Builder validation hints.
  • B. Background job not running: The publish is triggered via a scheduled job or flow. If the scheduler queue is backed up, the job's trigger has an unmet condition, or the job threw an unhandled exception on a previous attempt, it may not run again automatically. Check sys_trigger for job status and syslog for execution errors.
  • C. Scope conflict: The draft item belongs to a scoped application. The background publish job runs in the Catalog Builder's application scope. If the draft record or its child records are in a different scope and cross-scope read/write is not permitted, the publish job fails with a scope access error.
  • D. Concurrent edit collision: Two users editing the same draft simultaneously can create conflicting write states on the draft record.

Catalog Item Stuck in Draft State

If a user clicked "Save" instead of "Publish" — Draft is the correct state. If they submitted for publish and it stays in Draft: the publish trigger itself did not fire. Possible causes: custom business rule on the Builder draft table aborted the state transition; the user actually clicked Save, not Publish.

"Edit in Catalog Builder" Opens "Create New Item" Page

Root cause: When clicking "Edit in Catalog Builder" on an existing sc_cat_item, the Builder looks up the item's draft record by sys_id. If no draft exists (item was created/edited outside Builder), the Builder creates a new draft. A bug in some releases causes the Builder to fail to associate the new draft with the existing item, creating a fresh draft that renders as "Create new catalog item."

Diagnostic: Query the Catalog Builder draft table for the item's sys_id as a reference. If no draft exists, check whether the draft creation logic correctly set the item reference (a null item reference on the new draft causes the Builder to treat it as a brand-new item).

Cannot Create Question Choices for Variable via Catalog Builder

Root causes:

  1. Cross-scope restriction: The question_choice records must be in the same scope as the catalog item. If the item is in a custom scope but the user's session operates in the global scope (or vice versa), the write fails with a scope validation error.
  2. ACL on question_choice create: The catalog_builder_editor role may lack create permission on question_choice in some release configurations.
  3. Variable set scope mismatch: If the variable's variable set belongs to a different scope than the item, choice creation for that variable may fail scope validation.

catalog_builder_editor Cannot Create Record Producer from a Cross-Scope Template

Catalog Builder templates have a scope. When a catalog_builder_editor creates a new item from a template in a different scope, the platform checks that the user's current application scope can read the template via accessible_from settings on the template scope.

This is Working as Designed. Workarounds: elevate to catalog_admin for cross-scope operations; move the template to a globally accessible scope; have a catalog_admin create the item from the template, then assign it to the editor for further editing.

Catalog Builder: Loading Reference Table for New Fields Is Slow or Fails

Root cause: When adding a Reference-type variable in Catalog Builder, the Builder enumerates available tables via the sys_db_object table (all tables). On instances with thousands of custom tables or restricted ACLs on sys_db_object, this enumeration can take tens of seconds or fail entirely with a timeout.

Workaround: Use the backend item_option_new.do form to set the reference table directly, then return to Builder for the remaining variable configuration.

Template-Based Item Stuck in Review State

If a template is configured with a review step (approval workflow before publish), the item transitions to "In Review" after the editor clicks Publish. If no reviewer is assigned or the review notification was not sent (email configuration issue), the item appears stuck.

Diagnostic: Identify whether the template has a review/approval step configured. Check if the review notification was sent (syslog email entries or sys_email records around the publish time).

Known Issues — Catalog API & Flows

CartJS Not Returning Request Number After Checkout

Root causes:

  1. Asynchronous callback not waited: CartJS operations are asynchronous in the browser. Code reading the response synchronously sees an empty object — the request IS created, it just hasn't returned to the caller yet.
  2. Checkout validation failure: Mandatory variables not set before checkout cause the server-side checkout validation to fail and return an error response instead of a request number. Check the response's error field.
  3. Cart already empty at checkout time: If the cart was cleared between adding items and calling checkout (concurrent request from another tab), checkout succeeds with an empty cart and returns no request.

Duplicate RITMs Created Intermittently via CartJS

CartJS checkout is not idempotent — calling checkoutCart() twice on the same cart creates two requests. Causes:

  1. Double-click on submit (submit handler fires twice, button not debounced)
  2. Network retry — browser timeout on first checkout response, client retries, server processes both
  3. Script Include called twice — two business rules both invoking CartJS checkout on sc_cart_item insert

Public Users Unable to Submit Record Producers via API (403)

All three gates must be satisfied for public API submission: (1) catalog item public accessibility, (2) API endpoint authorization for anonymous access, (3) target table create ACL permitting the guest/API user.

Flow Action "Get Catalog Variables" Errors

  • Cannot find variable by name: If a variable exists on the item definition but was not created in the variable pool at RITM creation (sc_item_option_mtom has fewer records than expected), the action will not output that variable. The flow gets null.
  • Empty list output for MRVS when rows exist: If the variable_set field on sc_multi_row_question_answer records doesn't match the item's MRVS variable set (e.g., from a copy/clone operation that didn't update the variable set reference), the action finds no matching rows.
  • Variable named 'action': The "Get Catalog Variables" action encounters the same naming collision — the variable named action may conflict with Java method names in the variable API. Rename the variable.

Flow Action "Submit Catalog Item Request" Fails When Moved Between Instances

Root cause: The action stores a catalog item sys_id as a static input. When a flow is exported via update set from DEV to TEST, the DEV sys_id is embedded in the flow definition. If the same item exists in TEST with a different sys_id (created independently in each instance), the action's hardcoded sys_id does not match any item in TEST.

Service Fulfillment Steps Base Flow — cellNode is null

Most common failure in the Service Fulfillment Steps base flow. When the flow uses "Get Catalog Variables" to extract MRVS data, if an MRVS inner variable has no value (user left it blank), the cell node is null, causing ITableVariableCellNode.getVariableName() NullPointerException. Always null-guard MRVS variable cell reads in fulfillment flows.

Catalog Delivery Date Calculation Incorrect

Incorrect delivery dates from: business hours calendar mismatch, timezone offset error, or custom script logic error in the delivery_time field or business rule. Read sc_cat_item.delivery_time and identify the business rule or flow step responsible for setting sc_request.delivery_date.

Conversational Catalog Flow Getting Stuck After Upgrade

After an upgrade, VA topic flow definition or catalog submission API contract may change. If the VA topic was customized, the custom topic may no longer correctly call the catalog submission API. Common post-upgrade causes: API contract change, session context loss between conversation turns, NowAssist skill version mismatch.

Catalog Task Created to Wrong Assignment Group

SCTASK assignment group comes from: (1) flow action's assignment_group input, (2) business rule on sc_task setting assignment_group, (3) catalog item's default assignment_group field. Trace backward via audit trail on sys_audit for the assignment_group field on the SCTASK.

Known Issues — AI Agents & Now Assist

AI Agent Architecture (Catalog as Tool)

When a catalog item is added as a "Tool" to an AI Agent, the agent's execution model bypasses the client-side rendering layer entirely. No g_form, no catalog client scripts, no UI policies execute — the AI Agent bypasses these. Only server-side catalog scripts (in the RP's script field), ACLs, and variable mandatory/type constraints apply.

AI Agent Fails to Accept User Reference Variable Values

Root causes:

  1. LLM cannot resolve to a unique sys_id: Multiple records match the display value; agent cannot determine which one the user means
  2. Reference qualifier blocks the resolved sys_id: Even if valid in the reference table, the variable's reference_qual may restrict which records are valid in the agent's session context (which differs from a human user's session)
  3. Reference table not accessible to the agent: ACL blocks the agent's execution user from reading the table

AI Tool Unable to Return a Catalog Item (Search Fails)

  1. Item not indexed in AI Search — newly activated items take time to appear in the index
  2. User Criteria blocking the agent's user — agent executes under a specific service account that may not pass item criteria
  3. Item not marked as conversational — items must have "Make conversational" enabled to appear in AI Agent tool discovery
  4. Entitlement requirement mismatch — item filtered from the agent's accessible items

Conversational Catalog Items Not Completing in NAVA

  1. Unsupported variable type: If a mandatory variable uses an unsupported type (Macro, UI Page, complex reference), NAVA cannot complete the submission
  2. Variable named 'action': Naming collision causes variable extraction to fail silently
  3. "Autonomous" setting not checked: NAVA prompts for confirmation on every variable including pre-filled ones, creating a looping confirmation experience
  4. LLM misclassifying user response: LLM misinterprets a substantive answer as a confirmation ("yes"/"no"), skipping variable collection for that turn

Now Assist Catalog Generation Issues

Only a Few Questions Generated

Root cause: LLM generates variable count proportional to information density of the prompt. A one-sentence prompt produces 3-5 variables. Additionally, variables with types the platform cannot map to valid item_option_new.type codes are silently dropped, reducing final variable count below what the LLM suggested.

Improvement: Include specific field names, types, validation requirements, and audience descriptions in the prompt.

User Identity Field Returning Wrong User When Names Match

Root causes:

  1. HR tasks and on-behalf-of submission: gs.getUserID() returns the manager's sys_id, not the employee's, when manager submits on behalf of employee
  2. Duplicate name resolution: if two users share the same first and last name, display-value lookup may resolve to the wrong user record

AI Agent Intermittent System-Generated Errors

  1. LLM inference timeout: LLM provider did not return a response within the expected window
  2. Catalog submission API transient failure: Database timeout or lock contention — retrying a few minutes later typically succeeds
  3. Session state loss between conversation turns: If session expires between turns (long pause in conversation), state is lost and next turn fails with missing context

Known Issues — Approvals & Request Management

Approval Field Moving to "Requested" Without Any Flow or Rule Setting It

Root cause: sc_request.approval transitions from not requested to requested automatically via OOB business rules:

  1. "Cascade Request Approval to Request Item": Runs on sc_request insert/update. When an RITM linked to the request has approval = requested, cascades that state up to sc_request.approval.
  2. "Start Workflow": On sc_request insert. When the request is created, if any item in the cart has an approval workflow configured, the workflow starts and immediately sets approval state to requested as its first activity.

approval = requested means the platform has initiated the approval workflow — NOT that a specific human approver has been assigned yet. This is WAD.

OOB Business Rules Taking ~11 Minutes

Root causes:

  1. GlideRecord query without index: Business rule queries sc_req_item filtered by request = <req_sys_id> without an indexed column — full table scan on millions of RITM records
  2. Synchronous workflow start: "Start Workflow" rule initiates the workflow synchronously; if first activities (approval request creation, notification sending) are slow, the business rule appears to "take 11 minutes"
  3. Large number of items on one request: One checkout with many RITMs (OG with many items) — cascade rule runs once per RITM, each time querying back to the parent request

Approval Fulfillment Criteria Not Working for Case-Sensitive Data

Root cause: Approval criteria evaluate field values as strings. If criteria uses string comparison (e.g., state = approved) and the stored value has unexpected casing (e.g., Approved vs. approved), the condition fails silently. Most common in custom approval workflows using scripted conditions without case normalization.

Domain Mapping on RITM Where Catalog Item Is Not Domain Separated

Root cause: In domain-separated instances, sc_req_item.sys_domain is typically inherited from the requesting user's domain. If the catalog item (sc_cat_item) is not domain-separated (sys_domain = global), the RITM may have a domain set that the approver's domain configuration doesn't expect.

Approval routing depends on sysapproval_approver.sys_domain matching the approver's domain. If the RITM's domain causes the approval to be in the wrong domain, approvers cannot see or action it.

Email "Show Details" Not Working in Activity Stream for Approvals

Root causes:

  1. glide.email.read.active is false — email records may not be created
  2. The approval notification email's sys_email record was purged by the email purge policy before the user clicked "Show details"
  3. Activity stream item has a null or incorrect link to the sys_email record (known rendering bug in some releases)

Request Management Issues

Attachments Not Carrying Over to REQ/RITM After Submission

Failure modes:

  • Mode A (Upload timing race): User submits before attachment upload completes (upload is async from browser). Attachment lands in sys_attachment after cart item no longer exists.
  • Mode B (Attachment variable not correctly linked): Variable value is empty or contains invalid sys_id at checkout — copy logic skips it.
  • Mode C (ACL blocking copy): Custom ACL restricts create on sys_attachment for the submitting user's context. Attachment stays linked to cart item, cleared when cart is cleared.
  • Mode D (Business rule interference): Custom business rule on sc_cart_item or sc_req_item clears or ignores the attachment reference before copy logic executes.

Order Guide Not Using RITM-Level Approvals

OG approval behavior is controlled at the included catalog item level, not the OG level. "Cascade Variables" does NOT cascade approval configuration — each item's approval policy must be independently configured.

Estimated Delivery Field Not Populating

The field remains empty when: no delivery time configured (sc_cat_item.delivery_time is null/zero); business rule condition not met; business hours calendar missing; locale/timezone issue.

RITM / SCTASK Freezes When Opening

Root causes:

  1. Variable Editor rendering loop: many reference qualifier queries on form load while initializing each variable's UI control
  2. Large variable pool: 100+ variables creating 100+ sc_item_option records — Variable Editor loads all synchronously
  3. Corrupted MRVS records: null io references, duplicate node_index values in sc_multi_row_question_answer
  4. Business rule on sc_req_item or sc_task in infinite loop: rule fires on update, triggers the condition it checks

Known Issues — Client Scripts & UI Builder

Client Script Issues

g_form.setVariablesReadOnly() Not Working

Failure modes:

  1. Service Portal context: Not implemented in portal's injected g_form object in some releases — call silently does nothing
  2. Script scope: If client script runs in isolated scope (isolate_script = true), the g_form reference may not govern the visible form
  3. Called before form fully initialized: Variables that register after the call are not made read-only

Fix: Replace with g_form.setReadOnly('<variable_name>', true) calls for each variable individually in Service Portal.

GlideAjax Call Not Working After Upgrade

Root causes:

  1. Script Include scope change: moved to different application scope in upgrade; cross-scope calls fail
  2. client_callable flag removed: Script Includes must have client_callable = true; upgrade may have removed this setting
  3. Method name mismatch: Script Include class method name changed in upgrade

onChange Not Triggered in Virtual Agent

In the Virtual Agent conversation context, catalog variables are collected conversationally — there is no rendered form, and therefore g_form and the onChange event do not exist. This is WAD. Implement dependent variable logic server-side in the item's catalog script or in the VA topic flow's custom scripts.

g_modal.confirm() Not Working in Workspace

g_modal.confirm() is a Classic platform modal API. Not available in Agent Workspace/Now Experience (uses Seismic framework) or Service Portal. Call either silently fails (modal never appears, script continues as if user confirmed) or throws g_modal is not defined.

Alternatives: In Workspace: use a UI Action with the confirmation flag, or a custom modal component. In Service Portal: use Angular's $uibModal service. In both: validate server-side in an onSubmit business rule.

onSubmit Catalog Script Blocking Submission Unexpectedly

Common patterns causing unexpected blocking:

  1. An if/else branch falls through without an explicit return true — defaults to return undefined (falsy = blocking)
  2. Async operation blocking: GlideAjax call initiating before callback completes; browser appears hung during synchronous wait for async result
  3. Uncaught JavaScript exception: aborts the script mid-execution, may return undefined (falsy = blocking)

Diagnostic: Temporarily deactivate onSubmit scripts one by one. Open browser DevTools Console during form submission for JavaScript errors.

UI Builder Catalog Issues

Duration Variable Not Displaying Correctly in Custom Portal

Root cause: Duration-type variables store values as a duration string ([days] [HH:MM:SS]). If the Seismic component reads sc_item_option.value directly and renders it as a string, the user sees 1 08:00:00 instead of "1 day, 8 hours." getDurationValue() is a server-side API — a Seismic component must request a pre-formatted value from a server-side data resource.

Catalog Item Submit Not Working in Custom UI Builder Workspace

Root causes:

  1. Missing API call configuration: submit button triggers a UI event but no API call is made
  2. CSRF/Session token not included in API call: custom JavaScript fetch calls may not include the CSRF token (UIB data fetch components include this automatically)
  3. Scope permission on catalog submission API: UI Builder page's application scope must have access to the catalog submission API

Record in Workspace Keeps Loading (Infinite Spinner)

Root causes:

  1. Data resource query never resolving — a GlideRecord query, table API, or scripted data resource has a condition that never returns results or throws silently
  2. Circular dependency between components — two UI Builder components depend on each other's data outputs
  3. Large result set from an unfiltered data resource — page waits for all records before rendering

Systematic debugging of Service Catalog issues requires a set of 8 tools covering both server-side and client-side perspectives. Always start with a clear problem statement and an understanding of which layer (data model, variables, UI policy, script, flow) is involved.

  → Check catalog_ui_policy (ui_type, condition, order)

  → Check sc_multi_row_question_answer for MRVS integrity

Debugging Playbooks

Playbook 1: Variable Not Submitting

  1. Check if the variable appears in the form at all — if not, check catalog_ui_policy (visible=false)
  2. Check catalog_script_client with onSubmit type — temporarily deactivate to isolate blocking script
  3. After submission: check sc_item_option_mtom count vs expected. If missing variable: check syslog for exceptions at checkout time.
  4. Check item_option_new.write_roles — ACL may be blocking the variable value write

Playbook 2: RITM Not Created

  1. Check sc_cart and sc_cart_item — did the cart items exist at checkout time?
  2. Check syslog for exceptions during ProducerValve execution at checkout timestamp
  3. Check sys_flow_context for any error contexts created at checkout time
  4. Verify the catalog item's active = true and User Criteria passes for the submitting user

Playbook 3: Performance — Slow Ordering

  1. Check sys_transaction_log for the checkout transaction — identify which SQL query consumes the most time
  2. Evaluate reference qualifiers on the item's variables — each fires a separate query on form load
  3. Check the User Criteria cache state at /cache_inspect.do — if USER_ITEMS_CACHE is cold, criteria re-evaluate for every item
  4. Check catalog item cache (ZZ_CATALOG_ITEM_CACHE_YY) — if cold, item data is fetched from DB on each load

Playbook 4: Wrong Variable Value Stored

  1. Read sc_item_option.value directly for the affected variable — compare to what the user submitted
  2. Check for a before update business rule on sc_item_option that resets the value
  3. Verify the variable name does not collide with a g_form API method (e.g., action)
  4. For reference variables: confirm the sys_id (not display value) was submitted — OptionMap has no display_value → sys_id lookup

ATF Portal Timeout Issue

!

Common ATF failure: The ATF portal page open timeout defaults to 5 seconds (glide.atf.service_portal.steps.load_page.wait_timeout). If a catalog item page takes longer to load (many variables, slow reference qualifier lookups, cold cache), the ATF step times out before g_form is initialized. All subsequent variable steps fail with g_form is not defined. Fix: set the property to 15 or higher and re-run.

Widget Cloning Risk for ATF

Cloning and modifying the SC Catalog Item or SC Order Guide widget removes OOB ATF hooks. Any clone must replicate the OOB widget's g_form initialization code and ATF support calls exactly, or catalog behaviors (policies, scripts, validation, and ATF steps) will silently stop working.

Test Scenarios — Key Areas

  • Variable population: Verify all variable types save correctly (reference, select box, date, MRVS rows)
  • UI policy behavior: Verify visibility/mandatory/disabled state changes when trigger conditions are met
  • onLoad default values: Verify defaults are applied on form load before user interaction
  • Cross-experience parity: Test the same item in Classic, Service Portal, and Employee Center — UI policy ui_type mismatches surface here
  • Checkout integrity: After ordering, verify sc_item_option_mtom count matches item_option_new count for the item
  • MRVS row operations: Add, edit, and delete rows. Verify sc_multi_row_question_answer records are consistent after each operation.
  • Approval flow: Submit an item with approval configured. Verify sysapproval_approver is created and RITM stage transitions correctly.
  • Record producer target record: Verify the target table record is created with correct field values after submission.

100 First-Principles Questions & Answers

Every answer is grounded in source code from the locally cloned repos: app-common, glide, glide-services. These questions cover the deepest internals of the Service Catalog platform.

Category 1: Data Model Foundations (Q1–Q20)

Q1

Why does sc_cat_item extend sys_metadata while sc_req_item extends task?
sc_cat_item extends sys_metadata making it configuration data — update-set tracked, versionable, deployable between instances. sc_req_item extends task making it transactional data — with number, state, assignment, SLAs, and workflow inherited from the task table. If sc_cat_item extended task, catalog items would appear in task lists, consume task number sequences, and be subject to task ACLs. Update sets would not capture them as configuration.

Q2

Does sc_req_item have its own number series (RITM prefix) or does it share the task series?
sc_req_item has its own number series with the RITM prefix, defined via a sys_number table row. Although it inherits the number column from task, the number maintenance record points to a separate prefix — so RITMs get RITM000xxxx while tasks get TASK000xxxx.

Q3

The variable_pool column on sc_req_item is a GlideElementVariables type. Is it actually stored in MySQL or is it virtual?
variable_pool is a virtual GlideElement — it does NOT exist as a MySQL column. Registered via var_dictionary as type GlideElementVariables. At runtime, initialized with _setValue(VARIABLE_POOL), but the backing store is sc_item_option table (one row per variable per RITM) plus a serialized blob for fast reads. Calling g_form.getValue('variable_pool') returns empty string — individual variables are accessed as gr.variables.some_var.

Q4

What is the exact column name that sc_item_option uses to reference sc_req_item?
The connection is through the sc_item_option_mtom M2M table, not directly in sc_item_option. The M2M table has request_item referencing sc_req_item. sc_item_option itself has 8 columns: cart_item, sc_cat_item_option, item_option_new, value, order, generated_ai, plus inherited metadata. Using addQuery('request_item', ...) on sc_item_option generates a QueryEventLogger warning but still executes via the MTOM join — use addEncodedQuery('sc_request_item=...') for the correct column.

Q5

Can a variable (item_option_new) belong to both a catalog item AND a variable set simultaneously?
Yes, with a nuance. item_option_new.cat_item and item_option_new.variable_set are independent nullable FKs. A variable owned directly by a catalog item has cat_item set and variable_set null. A variable owned by a set has variable_set set and typically cat_item null. When a variable set is attached to a catalog item via io_set_item, the variables in that set are loaded alongside the item's direct variables — but the variable records themselves still reference the set, not the item directly.

Q6

sc_task has request_item (→sc_req_item) and also inherits parent from task. Are these the same field?
Different fields. sc_task defines its own request_item column as a reference to sc_req_item. The inherited parent field from task is a generic self-referential task parent that can point to any task subtype. In practice, sc_task.request_item is the correct FK to navigate from an sc_task to its RITM. sc_task.request points to the sc_request container.

Q7

How many valve classes extend ACatalogValve or ProducerValve?
4 concrete valve classes: ProducerValve (base for all ordering), CompositeProducerValve (extends ProducerValve for CRP multi-target), ScriptCatalogValve (extends ACatalogValve), and FacadeValve (extends ACatalogValve for save/preview). Dispatch: GlideappValveProcessor iterates registered valves, calls before() on each. First match wins — chain stops. CompositeProducerValve.before() additionally checks isNotCRP(item).

Q8

How many table name constants does ISharedCatalogConstants define?
ISharedCatalogConstants defines 40+ table/entity constants (lines 12–141). Key groups: SC request family (SC_REQUEST, SC_REQ_ITEM, SC_TASK, SC_CART, SC_CART_ITEM), catalog definition (SC_CAT_ITEM, SC_CATALOG, SC_CATEGORY), variable tables (SC_ITEM_OPTION, ITEM_OPTION_NEW, ITEM_OPTION_NEW_SET, IO_SET_ITEM), and newer additions like SC_DP_DATUM_TASK.

Q9

What type is sc_cat_item.checked_out — boolean, string, or reference?
checked_out is a boolean flag (constant CHECKED_OUT = "checked_out" in ICatalogConstants). During checkout: the published item gets checked_out=TRUE (state stays 'published'). The new draft gets checked_out=FALSE, state=draft. Cancel checkout: deletes the draft, resets checked_out=FALSE on published item. The state field tracks the lifecycle stage as a string enum (draft/published/in_review/publishing).

Q10

How is the relationship between sc_catalog and sc_cat_item stored — direct FK or M2M?
Stored as a glide_list (denormalized array) field directly on sc_cat_item. Column: sc_cat_item.sc_catalogs (plural) — type glide_list with array="denormalized". This is NOT a separate M2M table — multiple catalog sys_ids are stored as a list within the single column. One catalog item can appear in multiple catalogs without a join table.

Q11

Is the cart name 'DEFAULT' hardcoded? Can a user have multiple named carts?
DEFAULT is hardcoded as constant DEFAULT_CART_NAME in ICatalogConstants. When no cart name is passed, CartRecord defaults to it. Multiple named carts ARE supported — CartJS constructor and CartRecord constructor both accept a cartName parameter. Scripts can create custom-named carts by passing a non-null name to the API. The SYSPARM_CART_NAME parameter allows this from HTTP calls.

Q12

Does io_set_item carry any behavior configuration beyond the two FK references?
Yes — the order field. io_set_item has 3 columns: sc_cat_item (FK), variable_set (FK), and order (integer). The order field controls the sequence in which variable sets are displayed on the catalog item form — lower order appears first. This allows reordering shared variable sets differently per catalog item.

Q13

Does question_choice have a direct catalog_item reference, or only a question reference?
question_choice has only a question FK (→ item_option_new). There is no direct catalog_item reference. To find which catalog item owns a choice: question_choice.question → item_option_new.sys_id → item_option_new.cat_item → sc_cat_item. Or via variable sets: item_option_new.variable_set → item_option_new_set → io_set_item → sc_cat_item.

Q14

What is the type and all possible values for sc_req_item.stage?
Type: workflow (a choice/enum field). Exactly 3 possible values: waiting_for_approval (default), fulfillment (approved, fulfillment team working), delivery (being delivered to requester). Stage drives the RITM progress bar on the Service Portal order status page.

Q15

Is GlideElementVariables tightly coupled to the catalog, or is there an abstraction?
Tightly coupled to catalog, but with an abstraction layer. GlideElementVariables directly imports ISharedCatalogConstants, SC_ITEM_OPTION, SC_TASK, etc. However, it implements IGlideElementVariables interface and uses IVariablesComponent (line 76) via plugin registry — leaving a theoretical extension path for non-catalog tables. In practice, the current implementation only supports sc_req_item/sc_cart_item/sc_task family.

Q16

What exactly is sc_multi_row_question_answer's schema?
Table: sc_multi_row_question_answer (constant VARIABLE_NODE_TABLE). Columns: node_owner_table (parent table: sc_req_item, sc_cart_item, sc_task), node_owner_id (sys_id of parent record), node_variable_set_id (FK to item_option_new_set), node_question_id (FK to item_option_new), node_index (row number, 0-based), node_value (actual value), node_id (primary key).

Q17

What happens if publishItem() is called on an sc_cat_item already in 'published' state?
CatalogItemVersioningServiceJS.publishItem() calls CatalogItemPublishOperation.applyOperation(), which calls getStagingItem() first. That method only returns a record if the item is in draft, in_review, or published_checked_out state. If the item is in published state with no active checkout, getStagingItem() throws a CatalogItemVersioningOperationException. The caller receives an error JSON response — no re-publish occurs.

Q18

How many types of user criteria exist in the platform?
7+ types (all stored as glide_list fields on user_criteria): company, department, group, user, location, role, plus script (plain script) for arbitrary JavaScript evaluation. match_all (boolean) controls AND vs OR logic across types. advanced (boolean) enables advanced mode.

Q19

Does CartJS.submitOrder() use the same ProducerValve chain as the UI form submit?
Yes — same valve chain. CartJS.submitOrder() calls fCartRecord.submitOrder(map) (CartRecord.java), which calls checkoutCart(), which ultimately invokes the same ProducerValve → CompositeProducerValve path the UI uses. There is no separate code path for the API vs the form submit.

Q20

sc_item_option.order defaults to 1000000. What is its intended use?
The order field (default: 1000000) tracks display sequence of variables within a request item. The high default means unordered variables appear last. It is set during sc_item_option INSERT operations when variable values are persisted, matching the item_option_new.order value for that variable. In practice most implementations leave it at the default because sc_item_option is queried by question FK, not sorted by order.

Category 2: Variable System Internals (Q21–Q40)

Q21

For a MASKED variable (type 27), what exact transformation does HttpRequestVariableMapper apply?
Two-layer encryption via MaskedQuestion.encrypt(value): (1) KMF module cm_glide_sc_encrypter via GlideCryptoModule.getModule(); (2) Storage encryption via StorageEncrypter.getInstance().encrypt(value). Result format: $$%%encrypted_value%%$$. Stored in sc_item_option.value. Reading masked values requires calling MaskedQuestion.decrypt().

Q22

OptionMap.apply() for a Reference variable — what if display_value string is passed instead of sys_id?
OptionMap has no display_value → sys_id lookup for Reference variables. It calls ge.setValue(value) directly. If a display string like "John Smith" is passed instead of a sys_id, GlideElement.setValue() for a reference field will attempt to match it — this behavior is type/platform dependent and unreliable. Always pass sys_ids for Reference variables.

Q23

When is InternalOptionMap used, and what permission checks does it skip?
InternalOptionMap is used for internal/system catalog operations where normal user permission validation should be bypassed. It calls mapper.setIgnoreQuestionPermission(true) before parsing, which skips: q.canCreate() check (for new records) and q.canWrite() check (for updates). Annotated with: "Do not use this class without discussing with Catalog Team."

Q24

Question.getQuestionsMap() — does it use one SQL query or multiple?
Two SQL queries in sequence: First queries item_option_new for the bulk of questions. For any IDs not found there, a second query hits the core question table. Cache is checked first — if all IDs hit cache, zero queries fire.

Q25

For a DURATION variable (type 29), what exact normalization does HttpRequestVariableMapper do?
Creates a GlideDuration object, calls setValue(userValue) to parse from the user-provided format, then calls getValue() to get the normalized internal representation. This converts display durations (e.g., "5 hours", "1 day 3 hours") to the internal epoch-offset format.

Q26

MaskedQuestion — what encryption algorithm and where is the encrypted value stored?
Two-layer: (1) KMF module cm_glide_sc_encrypter via GlideCryptoModule; (2) StorageEncrypter for at-rest protection. Format: $$%%<encrypted>%%$$ where $$ is ENCRYPTION_IDENTIFIER and %% is STORAGE_ENCRYPTION_IDENTIFIER. Stored in: sc_item_option.value as the encrypted string. The plain text is never persisted.

Q27

AttachmentQuestion.copyVariableAttachment() — does it create a new attachment record?
Yes — creates a NEW attachment record. Reads from sys_attachment by the source sys_id, then calls SysAttachment.write() to create a new attachment on the target record. Tables: sys_attachment (read source) → sys_attachment (write new). The target table name is prefixed with USER_IMAGE_PREFIX.

Q28

GlideElementVariables.changes() — what conditions make it return true?
Returns true if the variable pool is loaded AND ANY of the following: any individual Variable.changes() returns true (compares against snapshot taken at load time); any container extension element's changes() returns true. If the variable pool was never loaded (isVariablePoolLoaded() = false), always returns false. Comparison is against a snapshot taken at load time, not a fresh DB query.

Q29

What is Variable.setSCItemOptionID() used for and when is it set?
Links a Variable object to the sc_item_option row that backs it. Set during loadVariablePoolQuestions() when variables are initialized from an existing RITM or cart item. The fSCItemOptionID field enables the variable to know which sc_item_option row to UPDATE when its value changes, rather than always INSERTing a new row.

Q30

Container Start (type 20) and Container End (type 21) — are they stored in sc_item_option?
No values are stored in sc_item_option for container variables. Container Start is a rendering grouping mechanism. Container End is a marker only that is pruned before rendering (its render() method is empty, getVisible() = false). They exist as item_option_new rows for definition purposes but never generate sc_item_option value rows.

Q31

GlideElementHierarchicalVariables — how does it differ from GlideElementVariables?
Extends GlideElementVariables with container parent-child awareness. Uses SequencedQuestionSet instead of a flat question list, preserving container nesting relationships. Used when the form needs to maintain container hierarchy (e.g., collapsible sections with child variables). Regular GlideElementVariables flattens all variables into a single list.

Q32

AnswerGenerator — what does it generate and when is it called vs OptionMap.apply()?
AnswerGenerator creates/updates question_answer table rows — an audit/history record of variable answers. It differs from OptionMap: OptionMap applies values to GlideRecord fields (creates sc_item_option rows), AnswerGenerator creates the audit trail in question_answer. The ai_generated flag marks AI-filled answers. Called during variable attachment operations, not during the primary OptionMap.apply() write path.

Q33

VariableSetSave — what is the 'slush bucket' pattern? What are lcodes_0 and lcodes_1?
A slush bucket is a two-list UI control: items move between "available" (lcodes_0) and "selected" (lcodes_1) lists. For catalog variable set assignment: lcodes_0 = available/inactive variable sys_ids; lcodes_1 = active/selected variable sys_ids (comma-separated). On save, all item_option_new rows for the catalog item are iterated; those in lcodes_1 get active=true, order=(index*100); others get active=false.

Q34

IVariablePoolQuestionSet — how many implementations exist and what differs between sc_req_item and sc_cart_item?
3 implementations. For sc_req_item: initialized via setRequestID(), loads variables from the request item context. For sc_cart_item: initialized via setCartID(), loads from the cart item. For sc_task: uses setTaskID(). Each loads from the same underlying tables but with different parent record references, affecting which sc_item_option rows are returned.

Q35

For TABLE_QUESTION (type 30 / MRVS), what HTTP parameter format is sent for multi-row data?
JSON Array format — one JSON object per row, each with question IDs as keys. Example: IO:variable_set_question_id = [{"q1_id": "iphone15", "q2_id": "alice_sysid"}, {"q1_id": "pixel9", "q2_id": "bob_sysid"}]. Parsing: JSONArray ja = new JSONArray(multiRowSetValue); for each i: JSONObject rowJson = (JSONObject) ja.get(i); then transform values per question type.

Q36

CatalogVariableUtilJS — what JavaScript API methods does it expose to catalog scripts?
Two static methods callable via CatalogVariableUtil.* from catalog scripts: removeUnusedAttachments(tableName, tableId) — removes all unreferenced attachments on the record; and clearOldAttachments(tableName, tableId, outdatedAttachmentIds) — removes specific listed attachments. Used in post-scripts and client scripts that manage file attachment cleanup after order submission.

Q37

sc_item_option.ai_assisted — when is it set to true?
Set to true when variable values are filled by NowAssist / conversational catalog AI. The AnswerGenerator.createQuestionAnswer() method accepts a boolean aiGenerated parameter that sets this flag. This allows fulfillment teams to distinguish user-typed values from AI-suggested values in the request item variables section.

Q38

Why is IOptionMap an interface rather than just using OptionMap directly?
Enables extension point injection — different implementations can be swapped via plugin registration. OptionMap (default, with permission checks) and InternalOptionMap (bypasses permissions) both implement IOptionMap. The plugin registry call PluginManager.newInstance(COM_GLIDE_VARIABLE, "option_map", IOptionMap.class) lets third-party plugins override the mapping behavior without changing catalog core code.

Q39

How and when is item_option_new.default_value applied — server-side render, client JS, or OptionMap.apply()?
Server-side during form render, NOT during OptionMap.apply(). The Question class calls evaluateDefaultInScope(gr, DEFAULT_VALUE) during Question initialization from the item_option_new record. The default is evaluated in the form's scope context (supports JavaScript expressions). By the time the form reaches OptionMap.apply(), the user has already seen and potentially changed the default value.

Q40

LookupSelectBox (type 19) vs Reference (type 8) — what are the key differences?
Both store a sys_id but differ significantly. Reference (8): Typeahead autocomplete UI, direct table lookup, simple qualifier. LookupSelectBox (19): Renders as SELECT dropdown, supports dependent lookups (fLookupDependantQuestion), pricing per choice (fLookupPrice, fRecurringLookupPrice), choice-source switching (fLookupSource = table OR choices), reference context (fReferenceContext). LookupSelectBox extends SelectQuestion, not ReferenceQuestion.

Category 3: Lifecycle & State Machine (Q41–Q60)

Q41

What are all valid values for sc_cat_item.state?
5 valid states in ICatalogConstants.java: draft (being edited, not visible), published (live and orderable), publishing (transitioning, brief intermediate), in_review (submitted for review before publishing), and checked_out flag = TRUE on a published item when a draft exists. The state field on the draft record is 'draft'. The published record keeps state=published with checked_out=true. They are TWO separate sc_cat_item rows.

Q42

CatalogItemCheckoutOperation — does checkout create a NEW sc_cat_item row or modify the existing one?
Creates a NEW sc_cat_item row (the draft). The published item is NOT modified for content — only its checked_out=TRUE flag is set. The new draft row gets: state=DRAFT, active=false, checked_out=FALSE, published_ref=<source_sys_id>. ALL columns are copied via GlideRecordUtil(gr).toMap(), excluding system fields.

Q43

Does CatalogItemPublishOperation copy variable data from draft to published?
ALL columns are copied via GlideRecordUtil(gr).toMap() during publish. Sequence: (1) Build draft→published question ID mapping; (2) Update question references across all linked tables (UI policies, client scripts, fulfillment stages); (3) Copy all fields from draft to published (sets state=PUBLISHED, checked_out=FALSE); (4) Handle translations; (5) DELETE the draft sc_cat_item row.

Q44

Cancel checkout — does it DELETE the draft or set its state?
DELETES the draft entirely. CatalogItemCancelCheckoutOperation calls CatalogItem.deleteCatalogItemAndReferences(checkedOutItem). The draft sc_cat_item row and all its associated variable definitions are permanently removed. The published item has checked_out=FALSE restored. There is no "cancelled" state — the draft simply ceases to exist.

Q45

ACatalogProcess — what infinite loop does it prevent and how?
Prevents recursive version increment loops. When CatalogUpdateEventGenerator.updateVersion() calls gr.update(), that update itself triggers CatalogDBListener.onExecute() again which would try to increment version again, ad infinitum. Prevention: before the version update, ACatalogProcess.markProcess() stores a transaction attribute key <recordId><operation>_process_marked. The skipEventGenerationForRecord() check reads this attribute — if marked, the event is skipped. The finally block calls unMarkProcess() to release the guard.

Q46-Q60 Summary: Event Management, Business Rules, Stage Transitions
Q46: CatalogEventManagerWrapper publishes CATALOG_ITEM_ENTITY_UPDATED events to ServiceNow event queue via EventManager.queue() to an empty-string custom queue (eventQueue="") to prevent blocking system events.

Q48: SCUtil.isPublishedItem() checks state == PUBLISHED; isPublishingItem() checks state == PUBLISHING. Used to guard version increments — only published and publishing items get version bumps.
Q49: Stage transitions driven by workflow activities and business rules, not ProducerValve. ProducerValve creates RITM with initial stage 'waiting_for_approval'. Workflow transitions move it to 'fulfillment' when approved, and to 'delivery' when tasks are closed.
Q50: ProducerValve.after() handles: post-script evaluation, redirect URL generation, and valve-level cleanup. sc_task creation is triggered by workflow/Flow Designer AFTER the RITM INSERT BR fires, asynchronously.
Q51: GlideappValveProcessor discovers valves via chain-of-responsibility pattern — iterates registered valve instances, each valve's before() inspects the HTTP action parameter (sysparm_action) and item type. First valve returning true claims the request.
Q52: OrderGuideSequencing.iterates through guide item rules (up to PAD_ACTIVITY_CREATION_LIMIT=20), calling createActivity(rule, laneID, order) for each item. Activities are numbered sequentially and represent catalog items within the guide.

Q54: ServiceCatalogCacheUtils caches individual sc_cat_item GlideRecord objects via cachedRecord(). Uses thread-safe ConcurrentHashMap with soft references (GC can evict under memory pressure). Invalidated by SYS_CACHE_FLUSH event, manual remove() calls, and ACatalogProcess markers.
