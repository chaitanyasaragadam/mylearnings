Quick reference

Use this mental model while watching or revising the demo.

Component
Declared through createCustomElement(tag, options).
View
Returns markup and receives state plus helper functions.
Properties
Inputs passed into the component with default, schema, required, reflect and computed behavior.
State
Runtime shape used to render and behave; updated immutably through updateState.
Actions
Events emitted with dispatch(type, payload, metadata, error).
Handlers
Effects that respond to emitted actions and optionally intercept or stop propagation.
Debugging chain: catalog data → properties → state computation → view render → dispatched action → action handler effect.

1. What this demo teaches

The recording explains how ServiceNow Service Catalog work is implemented in Seismic, the UI Framework used for Next Experience pages and builders.

  • Seismic is presented as a ServiceNow JavaScript framework built on Web Components standards.
  • The demo moves from framework concepts to practical Service Catalog screens like Catalog Builder, Create Question, fulfillment configuration, catalog item records, and related data tables.
  • The most important learning target is how a Seismic component is declared: tag name + options configuration.

2. Web Components foundation

Seismic builds reusable UI units using web component concepts. The demo highlights three important web-component ideas.

  • Lifecycle callbacks let the component react when it is inserted, rendered, changed, disconnected, or errors.
  • Shadow DOM gives encapsulation so outside styles and selectors do not accidentally affect the component internals.
  • Slots provide placeholders where consumer markup can be inserted, such as an icon next to button text.

3. Anatomy of a Seismic component

A UI Framework component is created with createCustomElement from @seismic/core. It receives two key arguments.

  • Tag: the DOM name of the web component.
  • Options: configuration object containing view, properties, actions, actionHandlers, renderer, state hooks, and behaviors.
  • The design encourages small reusable components whose behavior is driven by explicit configuration instead of hidden page logic.
createCustomElement('sn-first-component', {
  renderer: { type: snabbdom },
  view,
  properties,
  actions,
  actionHandlers,
  setInitialState,
  transformState,
  behaviors
});

4. View function

The view function returns the markup that Seismic renders into the DOM.

  • The view function receives state and helpers.
  • State represents persisted and computed state for the component instance.
  • Helpers expose dispatch, updateState, and updateProperties so the component can emit actions, update its state, or update component properties.
const view = (state, { dispatch, updateState, updateProperties }) => {
  return <div>Hello {state.properties.name}</div>;
};

5. Properties and state pipeline

Properties are the mechanism for passing data into components. State combines property values, defaults, persisted values and computed values before rendering.

  • default gives a property a fallback value.
  • schema describes the expected JSON type and supports development warnings.
  • required marks a property as required during development validation.
  • reflect maps a property value to an HTML attribute.
  • computed properties derive values from current properties/state.
  • transformState shapes computed state before it reaches the view.
properties: {
  firstName: { default: 'John', reflect: true },
  lastName: { default: 'Doe' },
  name: {
    computed({ properties: { firstName, lastName } }) {
      return `${firstName} ${lastName}`;
    }
  }
}

6. Actions and action handlers

Actions are the eventing mechanism for notifying the component, ancestor components, or the application that something happened.

  • dispatch emits actions with type, payload, metadata and optional error flag.
  • Actions bubble upward through the DOM tree.
  • Ancestors can observe a child action; siblings and child components do not automatically observe it.
  • Action handlers react to actions through effects, interceptors, modifiers and optional stopPropagation.
dispatch('ACTION_NAME', {
  /* payload */
}, {
  /* metadata */
}, {
  error: false
});

actionHandlers: {
  'SOMETHING_HAPPENED': (coeffects) => {
    // effect logic
  }
}

7. Lifecycle actions

The framework emits lifecycle actions to expose major component lifecycle moments.

  • COMPONENT_CONNECTED fires when an element is appended into a document-connected element.
  • COMPONENT_BOOTSTRAPPED fires once after initial connection and bootstrapping.
  • COMPONENT_DOM_READY fires after connection and rendering.
  • COMPONENT_PROPERTY_CHANGED fires when a property is set after initial bootstrapping.
  • COMPONENT_RENDER_REQUESTED and COMPONENT_RENDERED wrap the render cycle.
  • COMPONENT_DISCONNECTED and COMPONENT_ERROR_THROWN cover teardown and runtime errors.

After the concept slides, the demo switches into documentation, source code, and runtime inspection.

9. Service Catalog demo flow

The demo validates Seismic behavior in Service Catalog authoring screens.

  • Create Question wizard shows question type, subtype, label, tabbed sections, and live question preview.
  • Catalog Builder for Standard Laptop shows fulfillment configuration and a selected Flow Designer flow.
  • Questions section shows existing groups, inserted test question, and add/insert controls.
  • Classic record and table views are inspected to connect the Seismic builder UI back to catalog item data.

10. Code review section

The final section of the demo returns to implementation details in the codebase.

  • The code view is used to trace where Service Catalog UI components are declared.
  • The practical way to learn from the recording is to map each UI behavior back to component options: properties, view, actions and handlers.
  • When debugging, follow the chain: catalog data → component properties → state computation → view render → dispatched action → action handler effect.
