Learning path

Learn it in this order

The clean mental model is: catalog definition → cart capture → request execution → answer storage → fulfillment → cache refresh.

Step 1

Catalog surface

Start with what the requester sees: catalog homepage, categories, top requests, cart, and item detail pages.

Step 2

Item definition model

Move behind the UI to `sc_cat_item` and its related/extended tables: normal items, record producers, order guides, content items, and wizards.

Step 3

Variables and answers

Understand the split between variable definitions and submitted answer rows.

Step 4

Request lifecycle

Follow the submitted order into request, requested item, and catalog task records.

Step 5

Record producers

See how catalog inputs can create a non-catalog target record, such as Incident.

Step 6

MRVS/table variables

Read multi-row answers as row-index grouped cells rather than a single field.

Step 7

Behavior and policy

Locate where catalog UI behavior is configured through UI policies and related actions.

Step 8

Caching internals

Connect catalog configuration changes to cache flushes and source-level cache classes.

Cheat sheet

Tables you should recognize

Table / areaRoleHow to remember it
`sc_catalog`Catalog containerHolds catalogs such as Service Catalog; categories and items belong under it.
`sc_category`Category hierarchyGroups catalog items; can have a parent category such as Hardware → Tablets.
`sc_cat_item`Base catalog itemCentral item table. Normal items, content items, record producers, order guides, and wizard items are connected to this model.
`sc_cat_item_category` / related category M2MItem-category linkConnects catalog items to categories so they appear in the right catalog location.
`item_option_new`Variable definitionDefines catalog questions/variables: type, question text, name, order, catalog item, variable set, reference, default, and visibility rules.
Variable set tablesVariable groupingReusable sets of variables; multi-row variable sets become table-like inputs.
`sc_cart_item`Pre-checkout cart rowRepresents an item in the shopping cart before checkout.
`sc_item_option`Submitted answer valueStores each variable answer row, including question reference, value, order, and catalog option metadata.
`sc_item_option_mtom`RITM-to-option joinLinks requested items to the answer option rows.
`sc_request`Request containerThe overall order/request that can contain one or more requested items.
`sc_req_item`Requested item / RITMThe ordered item instance with lifecycle fields such as stage, state, quantity, and item reference.
`sc_task`Fulfillment taskWork record created to fulfill the requested item.
`question_answer`Record producer answersStores answers captured by record producers against the produced target record/table.
`item_produced_record`Producer-output mappingMaps the record producer to the target record/task it created.
`sc_multi_row_question_answer`MRVS/table-variable storageStores each cell of a multi-row variable set; rows are reconstructed using `row_index`.
`catalog_ui_policy` / actionsCatalog form behaviorControls visibility, mandatory, readonly, and dynamic behavior of variables.
`sys_cache_flush` + catalog cachesCache invalidationShows catalog/cache entries invalidated when item/category/configuration data changes.

Storage patterns

Where the values go

Normal catalog item

sc_cat_item defines the item. item_option_new defines variables. During ordering, sc_cart_item and sc_item_option hold item/answer data. After checkout, the request is represented by sc_request and sc_req_item.

Record producer

The user fills a catalog-like form, but the output is a target task/table record. The demo shows Create Incident, the Incident Variable Editor, question_answer, and item_produced_record.

Multi-row variable set

The visible grid is reconstructed from rows in sc_multi_row_question_answer. Same row_index means the cells belong to the same logical table row.

Advanced internals

Why the demo goes into code and cache

Catalog rendering depends on category, item, criteria, question, UI policy, and variable metadata. The code/cache portion explains why changing catalog configuration often triggers catalog cache invalidation.

Catalog/category cache

Category membership and item visibility are cached so the catalog homepage loads quickly.

Question cache

Question, choice, and regex metadata are cached because every catalog item render needs variable definitions.

Stale cache listeners

When certain tables update, the listener/finder logic determines which cached catalog data should expire.

Configuration views

Category and builder views

The demo also shows category membership and the guided Catalog Item Builder/Wizard path.

Practice

Self-check questions

Question 1

A user submits Standard Laptop with two software checkboxes. Which tables would you inspect to find the item, RITM, and answers?

Answer: sc_cat_item, sc_req_item, sc_item_option, and usually sc_item_option_mtom.

Question 2

A record producer creates an Incident. Why might the answers not be on sc_req_item?

Answer: Record producers create target records; answers can be traced through question_answer and item_produced_record.

Question 3

In MRVS, why do several records have the same row index?

Answer: Each record is one cell; same row index groups cells into one logical table row.
