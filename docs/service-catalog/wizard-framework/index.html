How to use this guide

Read it as a build map, not just notes

Start with the data model, then follow the build sequence, then use the implementation internals and checklist when you need to reproduce the pattern in another catalog flow.

Module

1. What this demo is about

The recording explains how Service Catalog Wizard framework turns a catalog item or composite record producer into a guided multi-step experience. The sample used throughout is Open Bank Account.

  • A wizard is not just a form. It adds navigation, sections, progress, feedback, actions, templates, and submission flow around a producer.
  • The demo starts from the UX anatomy, then configures a producer, wizard record, wizard steps, wizard sections, and questions.
  • The later part shows how the wizard is rendered in UI Builder and how source code, plugin packaging, submit handlers, template handlers, and declarative actions support the runtime behavior.

Module

2. Core data model to remember

The central chain is Producer → Catalog Wizard → Wizard Steps → Wizard Sections → Wizard Questions. Templates and action configurations are optional but powerful extensions.

  • Composite Record Producer / Producer Set: defines the actual catalog intake object and the target records/tables.
  • Catalog Wizard: the wrapper record that selects the producer, layout, title expression, save/review behavior, and feedback panel behavior.
  • Catalog Wizard Step: left-side navigation unit such as User Details, Account Details, Referrer Details, Review and Submit.
  • Catalog Wizard Section: visual group inside a step, usually rendered as a card or full-width section.
  • Catalog Wizard Question: mapped catalog question or variable displayed inside a section.
  • Wizard Action Configuration: actions shown in the wizard header/footer or tied to events, such as populate details or view details.
  • Catalog Template and Catalog Template Detail: hold reusable defaults, read-only flags, restrictions, and mandatory overrides.

Module

3. Build flow used in the demo

The demo repeatedly moves between the ServiceNow classic configuration forms, UI Builder, and runtime preview to prove that configuration changes are reflected in the final wizard.

  • Create or open the producer for the catalog item: Open Bank Account.
  • Create a producer set such as Create Bank Account, then define target table and variables.
  • Create the Catalog Wizard record and attach it to the producer.
  • Define wizard steps in the correct order and decide whether a review step is shown.
  • Create wizard sections under each step, then map catalog questions into those sections.
  • Open the UI Builder page and add the Catalog Wizard component.
  • Select the wizard record in component properties and tune mode, spacing, template, navigation button behavior, and event mappings.
  • Preview the page as an end user and validate navigation, required fields, templates, actions, save, and submit.

Module

4. How templates fit into the wizard

Templates let an admin or template creator prefill questions, enforce mandatory values, restrict values, or mark fields read-only for item owners.

  • sc_template stores the high-level template record: name, short description, availability, and catalog item reference.
  • sc_template_detail stores field-level rules: template reference, question/item option reference, value, mandatory, read-only, and restricted settings.
  • Templates can be applied while creating new wizards and also to existing wizards.
  • The mandatory override screen demonstrates how template owners can force an item owner to edit chosen values before publishing or submitting.

Module

5. Runtime user experience

The final wizard gives the user a structured intake page with step navigation, card-based sections, validation, action buttons, and submit controls.

  • The left rail shows step navigation and current position.
  • Cards split each step into logical sections such as Name, Details, Account Type, Branch Details, Referrer, and Details.
  • Footer buttons move users forward and backward, while header buttons can expose save, submit, and custom actions.
  • The sample shows populated branch data and referrer data, proving that variables, actions, and mappings are wired together.

Module

6. Implementation internals shown

The demo does not stop at configuration. It opens the repository and Java/backend handlers to show how the wizard component is implemented.

  • The frontend package contains connected components, action handlers, data fetchers, submit handlers, interceptors, views, styles, and utilities.
  • The component exposes properties for wizard data, loading state, success state, selected wizard, template, and error message.
  • Submit handlers create the payload, control dirty/submitting state, invoke validation, and dispatch save/submit actions.
  • Backend Java handlers call the wizard processor layer for submit, save, and action access checks.
  • Declarative actions and UI Builder event mappings connect custom buttons to client scripts and modal/action behavior.

Build checklist

Practical implementation checklist

Use this when implementing a new wizard or debugging why an existing wizard does not render correctly.

  1. Confirm the producer/record producer opens correctly before wrapping it in a wizard.
  2. Keep the wizard step order simple and user-facing: who, what, details, review.
  3. Group questions into sections that match how the end user thinks, not how tables are stored.
  4. Use the review step only when the request has enough complexity to justify review before submit.
  5. Validate template behavior separately: prefill, mandatory, restricted, and read-only scenarios.
  6. Test Save and Submit independently; save can succeed while submit can still fail validation.
  7. Map custom actions in UI Builder and verify that handled events appear as expected.
  8. Test with dirty forms, missing mandatory values, and data fetched by reference scripts.
