Learning Summary

The demo shows how a ServiceNow Service Catalog Wizard is configured on top of a composite record producer. The main concepts are the producer, producer set, catalog wizard, wizard steps, wizard sections, wizard questions, template records, UI Builder catalog wizard component, submit/action handlers, and declarative action/event mappings.


Key Configuration Objects

- **Composite Record Producer** — Main producer/catalog intake record. Example in demo: Open Bank Account.
- **Producer Set** — Target-record definition and variable grouping. Example in demo: Create Bank Account mapped to Bank Account table.
- **Catalog Wizard** — Top-level wizard wrapper around a producer. Example in demo: Open Bank Account wizard with stacked layout.
- **Catalog Wizard Step** — Step shown in left navigation. Example in demo: User Details, Account Details, Referrer Details, Review and Submit.
- **Catalog Wizard Section** — Visual section/card within a step. Example in demo: Name, Details, Account Type, Branch Details.
- **Catalog Wizard Question** — Question/variable displayed inside a section. Example in demo: First Name, Last Name, Account Type, Branch Code.
- **Catalog Template** — Reusable template header. Example in demo: sc_template.
- **Catalog Template Detail** — Field-level template rule. Example in demo: sc_template_detail.
- **Wizard Action Configuration** — Action rendered or triggered in the wizard. Example in demo: Populate Referrer Details, View Referrer Details.
- **UI Builder Catalog Wizard component** — Runtime component placed on a page. Example in demo: Component properties and event mappings.


Practical Checklist

- Confirm the producer/record producer opens correctly before wrapping it in a wizard.
- Keep the wizard step order simple and user-facing: who, what, details, review.
- Group questions into sections that match how the end user thinks, not how tables are stored.
- Use the review step only when the request has enough complexity to justify review before submit.
- Validate template behavior separately: prefill, mandatory, restricted, and read-only scenarios.
- Test Save and Submit independently; save can succeed while submit can still fail validation.
- Map custom actions in UI Builder and verify that handled events appear as expected.
- Test with dirty forms, missing mandatory values, and data fetched by reference scripts.
