1. Command Processor Framework

Package: com.glide.catalog.processor.command

The processor layer implements a Command pattern for dispatching AJAX and HTTP requests to the appropriate catalog operation handler.

ICommand (Interface)

Object execute(ICommandContext context)

All command implementations receive a context object and return a result.

ICommandProcessor (Interface)

ICommand resolve(String action)  // finds the right command for an action name
void process(ICommandContext context)

Command Registry

Commands are registered by action name. Processor resolves the action string from the HTTP request → finds the command → executes it.

HTTP Adapter

Bridges servlet HTTP requests into the command framework: - Extracts action parameter from request - Builds ICommandContext with request/response - Delegates to ICommandProcessor.process()

Key Command Implementations (processor/command/impl/)

  • Cart operations: add, remove, update, checkout
  • Catalog browsing: item details, category listing
  • Variable operations: get/set variable values
  • Policy evaluation: on-change/on-load triggers

2. AJAX Processors

Package: com.glide.catalog.processor + com.glideapp.servicecatalog (root)

These are servlet-style processors handling specific AJAX operations from the catalog UI.

CartAjaxProcessor / CartAjaxProcessorV2

Purpose: Handle all cart AJAX operations.

Action Description
addToCart Add item to cart, return updated cart JSON
updateCart Update existing cart item (quantity, variables)
removeFromCart Remove item by cart_item sys_id
checkoutCart Submit cart, return request sys_id
getCart Return current cart contents as JSON
orderNow Add + immediately checkout

V2 differences: Improved error handling, JSON response envelope, better domain validation.

CatalogSuggestProcessor

Purpose: Type-ahead/autocomplete suggestions.

  • Input: partial text + catalog context
  • Output: JSON array of matching catalog items/categories
  • Respects user criteria and domain

RequestHintProcessor

Purpose: Return helpful hints for request fields.

  • Returns hint text for specific question/field combinations
  • Used in forms to guide user input

LookupSelectProcessor

Purpose: Reference field lookup for variable questions.

  • Input: reference table, qualifier, partial text
  • Output: matching records for dropdown population
  • Evaluates reference qualifiers in variable context

CurrencyAjaxProcessor

Purpose: Real-time currency conversion for price display.

  • Input: amount, source currency code
  • Output: converted amount in session currency
  • Uses GlideElementCurrency and Converter

3. NG UI Layer

Package: com.glide.ui.ng.catalog

The NG (Angular/Next-Gen) UI layer provides API endpoints consumed by the ServiceNow portal and workspace interfaces.

NGCatalogClientScriptProcessor

Purpose: Load client scripts (ON_LOAD, ON_CHANGE, ON_SUBMIT) for a catalog item or variable set in NG UI context.

Returns: Structured JSON of scripts filtered by: - Script type (on_load, on_change, on_submit) - Record type (sc_cat_item, sc_req_item, sc_task, producer target) - Domain applicability

SC Model Classes (sc/model/)

Data transfer objects (DTOs) for NG UI responses:

Class Data Carried
SCCatalogModel Catalog metadata for NG portal
SCCategoryModel Category metadata + visibility
SCItemModel CatalogItem data for item detail view
SCCartModel Cart contents + pricing for NG cart
SCRequestModel Request confirmation data

Utility Classes (util/)

Helpers for transforming domain objects into NG-consumable JSON structures.

4. Data Lookup System

Package: com.glide.data_lookup.catalog

Data lookups are table-driven auto-population rules. When a user fills a specific variable, data lookup queries a related table and pre-fills other variables.

CatalogDataLookup

Purpose: Evaluates data lookup rules for a catalog item and applies matching values.

Flow:

1. Load data lookup rules for catalog item
2. For each rule:
   a. Evaluate condition: does the trigger variable match?
   b. If match: query target table with field conditions
   c. Apply matching record values to destination variables
3. Return map of {variableName: newValue}

Caching: Lookup rules cached per catalog item sys_id.

GraphQL Data Lookup (catalog/graphql/)

Extends data lookup capabilities with GraphQL-compatible query structures.

Script API (catalog/script/)

Exposes data lookup operations to server-side Rhino scripts for custom rule evaluation.

5. GraphQL Schema

Package: com.glideapp.catalog.graphql.schema

Defines the GraphQL types and queries for catalog data access (used by portal and mobile).

Key Types

GraphQL Type Maps To Key Fields
CatalogItem sc_cat_item id, name, shortDescription, price, active, canOrder
Category sc_category id, title, description, items, subCategories
Catalog sc_catalog id, title, categories, items
Variable item_option_new id, name, label, type, mandatory, choices
VariableSet item_option_new_set id, variables, layout
CartItem sc_cart_item id, catalogItem, quantity, variables

Security Definitions

  • Every query/field respects ACL enforcement
  • User criteria evaluated at resolver level
  • Domain filtering applied via DomainSupport
  • canOrder() / canView() checked before returning items

Query Types

type Query {
  catalogItem(id: ID!): CatalogItem
  category(id: ID!): Category
  catalog(id: ID!): Catalog
  searchItems(term: String!, catalogId: ID): [CatalogItem]
}

6. Streaming

Package: com.glide.catalog.stream

SysId2ObjectProducerStream

Purpose: Lazy-evaluation stream over a list of sys_ids, producing domain objects on demand.

Pattern: Supplies objects one-at-a-time rather than loading all into memory. Useful for large catalog/category item sets.

Interface: IStream<T>

boolean hasNext()
T next()
void close()

Use cases: - Streaming category items for large catalogs - Large-scale search result iteration - Cart item enumeration without full list materialization

7. Global Search

Package: com.glideapp.servicecatalog.globalsearch

ServiceCatalogGlobalSearchConfig

Purpose: Configures ServiceNow's global search to index catalog items.

Indexed fields: - Item name (name) - Short description (short_description) - Description (description) - Keywords

Security: Index only includes items the user can view (user criteria evaluated at index time).

Search config registers: - Table: sc_cat_item - Fields to index and rank - User criteria filter for personalized results

8. AI Search

Package: com.glideapp.servicecatalog.aisearch

AisCatalogItemIndexer

Purpose: Indexes catalog items for AI-powered search (NLU/semantic search).

Indexing behavior: - Extends ServiceCatalogGlobalSearchConfig with semantic metadata - Includes: item name, short description, keywords, category names - Security-bound criteria: Index respects active status + user criteria at index time - Registers index schema for AIS (AI Search) engine

Difference from global search: - Global search: keyword/field matching - AI search: semantic/NLU-based similarity matching

9. Tags

Package: com.glideapp.servicecatalog.tags

Purpose: Manages catalog item tagging for organization and filtering.

Key operations: - Tag a catalog item with a label - Search/filter catalog items by tag - Display tag chips in catalog UI

Backed by: Platform tagging infrastructure (label_entry / label tables).

10. Standard Ticket Configuration

Package: com.glideapp.servicecatalog.standard_ticket

StandardTicketConfig / StandardTicketConfigCache

Purpose: Configuration-driven ticket creation. Defines default field values for tickets created from catalog items.

StandardTicketConfigCache: - Cached per catalog item sys_id - Cache invalidated via CatalogDBListener on config changes

Operations: - Load standard ticket configuration for an item - Apply default field values to new ticket/request records
