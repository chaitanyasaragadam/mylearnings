Concept

What is a Table Variable / MRVS?

A table variable is ServiceNow’s way to collect repeating, row-based information while a user orders or submits a catalog item. It is also called a Multi Row Variable Set.

Simple example: ordering multiple mobile devices. One request may contain several rows, and every row has the same columns: Device Type, Storage, Color, Quantity.
  • Use it when the user must enter multiple similar records.
  • Use child variables as the table columns.
  • Use row index / answer grouping to understand how values are saved.
  • Test in the exact UI surface your users will use: Platform, Portal, or Workspace.

End-user behavior

How it appears to the requester

In the order form, the MRVS is rendered as a table with an Add action. Clicking Add opens a modal. The modal contains the child variables for one row.

After the user adds a row, the row appears in the grid. Users can add multiple rows, edit rows, or remove rows depending on UI behavior and permissions.

For usability, keep the MRVS columns focused. Too many columns make the row modal and grid harder to use.

Persistence model

Where MRVS values are stored

The demo opens the Multi Row Question Answers list to show how row data is persisted. Think of each table cell as an answer record, and the row index ties multiple answers together into one row.

Storage mental model

  1. The MRVS itself defines the table shape.
  2. Child variables define the columns.
  3. During ordering, values may be tied to cart-level records.
  4. After submission, values can be tied to requested-item records.
  5. The row index groups individual question answers into one grid row.

Implementation architecture

How the same MRVS renders in different UI surfaces

The demo separates implementation details into Platform, Portal, and Workspace.

Platform

  • Data cache: window.tableVariableCache
  • Templates: question_table.xml, question_table_variable_template.xml, question_table_variable_row.xml
  • APIs: TableVariableService.js, TableVariable.js

Service Portal

  • Templates: sp_form_field.xml, sp_element_sc_multi_row.xml
  • Widget: sc-multi-row-active-row
  • Directive: spScMultiRowElement.js

Workspace

  • Component: <sn-catalog-form-multi-row-variable>
  • GraphQL fragment/type: AppCatalog_MultiRowVariableSetType
  • Builder: QuestionElementType.buildMultiRowVariableSetType()
  • Widget call: sc-multi-row-active-row

Developer walkthrough

Portal directive behavior

The code walkthrough shows the directive controlling row display, clearing values, and opening the active-row modal.

  • refreshMultiRowDisplayValue() refreshes the display value for the MRVS.
  • clearAllValue() clears the full grid value and sends a confirmation/status message.
  • clearValue() asks for confirmation and removes all rows.
  • createRow() opens the Add Row flow using active-row widget data.

Workspace

Requested Item display

In Workspace, the requested item form displays the Variables section and renders the MRVS grid. The demo shows submitted rows with values such as Apple iPhone 8, 64GB, Black, and quantity values.

Do not assume Portal and Workspace use identical rendering paths. Test MRVS behavior in all supported surfaces.

Self-check

Quick quiz

1. What makes a Variable Set behave like a table?

Its Type is set to Multi Row.

2. What do child variables become inside an MRVS?

They become the columns/questions shown in the Add Row modal and grid.

3. How are multiple cell values grouped into one logical row?

By the row index in the Multi Row Question Answers data.

4. Why should you test in Portal and Workspace separately?

They use different rendering components/templates/directives even though the catalog concept is the same.
