1. Valve Architecture Overview

The Valve pattern (Chain of Responsibility) is the core request-processing architecture. When a user submits a catalog item, the HTTP request flows through a ValveProcessor that dispatches to the appropriate valve chain.

HTTP Request
    ↓
ValveProcessor
    ↓
ACatalogValve.init(request, response)
    ↓
ACatalogValve.before()  ← valve-specific logic
    ↓ (depends on item type)
  CartValve
  ProducerValve        ← record producers
  CompositeProducerValve ← multi-record producers
  ScriptCatalogValve   ← script execution

2. ACatalogValve (Abstract Base)

Package: com.glideapp.servicecatalog.valve
Extends: Servlet-based HTTP request handler

Contract:

void init(HttpServletRequest req, HttpServletResponse resp)
void init(HttpServletRequest req, HttpServletResponse resp, Object context)
abstract void before()   // subclasses implement processing logic

All valves extend this class and implement before().

3. ProducerValve

File: ProducerValve.java (1211 lines)
Extends: ACatalogValve

Purpose: Orchestrates creation/update of a target ITSM record (incident, change, task, etc.) from a catalog item of type RECORD_PRODUCER.

Key Fields

Field Purpose
fRedirect Post-production redirect URL
fPortalRedirect Portal-specific redirect
fTableName Target table (e.g., incident)
fProducedRecordId sys_id of created/updated record
fTargetRecord GlideRecord for edit mode
fHttpMapper HTTP parameter → variable mapper
fEdit Edit mode flag (update existing vs. create new)
fSequence Order guide sequencing metadata
fCartItem Cart/request item being produced

5-Step Production Pipeline

Step 1 — newrecord
  Create GlideRecord for target table

Step 2 — apply template
  CatalogTemplateManager.apply(template, targetRecord)

Step 3 — merge variables
  If edit mode: ProducerTargetVariableMapper extracts from target record
  If new: HTTP parameters mapped to variables

Step 4 — run user script chain
  SaveScriptEvaluator
    → PreScriptEvaluator
    → MacroSaveScriptEvaluator
    → database insert/update
    → PostScriptEvaluator
    → MacroPostScriptEvaluator

Step 5 — log produced record
  sc_item_produced_record updated with target record sys_id

Key Methods

Method Purpose
before() Entry point, delegates to produceItem()
produceItem(boolean) Full production with optional cart handling
produceItem(Map<String,String>) Production with explicit parameter map
applyTemplate() Apply CatalogTemplate values
evaluateScript() Run script evaluator chain
getTargetRecord() Retrieve existing record for edit mode
getProducedRecordId() Return sys_id of produced record
produceItemReturnTargetRecord() Used by OrderGuideSequenceFulfillment

4. CompositeProducerValve

File: CompositeProducerValve.java (300 lines)
Extends: ProducerValve

Purpose: Handles catalog items that produce multiple target records simultaneously (e.g., create an incident AND a change together). Adds rollback/transaction safety.

Additional Fields

Field Purpose
fRollbackContext Transaction rollback manager
fRollbackRecorder Logs changes made for rollback
fProducerSetMap Maps producer sets to generated records

Key Methods

Method Purpose
before() Override: sets up composite context, then calls super.before()
generateProducerSetTargetRecords() Create records for each ProducerSet configuration
insertOrUpdateRecords() Persist all records to DB
deleteRemovedTargetRecords() Remove records deleted through UI

Composite Flow

1. Parse JSON arrays from request (multi-row variable sets)
2. Create RollbackRecorder
3. For each ProducerSet:
   a. ONE_TO_ONE → create single target record
   b. Multi-row → parse JSON array → create one record per row
4. Call parent ProducerValve.before()
5. On failure → RollbackContext.rollback() removes all created records

5. CartValve

Extends: ACatalogValve
Purpose: Pre/post hooks for cart operations (add, update, remove, checkout). Executes business rules around cart state changes.

6. ScriptCatalogValve

Extends: ACatalogValve
Purpose: Executes custom Rhino scripts at valve checkpoints via GlideScopedEvaluator.

7. Script Evaluator Chain

Package: com.glideapp.servicecatalog.valve.script_evaluators

RecordProducerScriptEvaluator (Abstract Base)

abstract boolean doEvaluate()       // subclass-specific logic
abstract boolean shouldReturnError() // whether to propagate exceptions
boolean evaluate()                   // executes doEvaluate(), catches exceptions
void putAdditionalProps()            // add macro/question context to Rhino scope

Rhino context setup: - fGlideScopedEvaluator — wraps target record and script record as Rhino globals - fTargetRecord — the record being produced - fScriptRecord — the catalog item / question record containing the script

Evaluator Instances

Evaluator Script Field When Executed Propagates Error
SaveScriptEvaluator SAVE_SCRIPT Before DB insert Yes
PreScriptEvaluator SCRIPT (pre-insertion) Before DB insert Yes
MacroSaveScriptEvaluator SAVE_SCRIPT on question Before DB insert Yes
PostScriptEvaluator POST_INSERTION_SCRIPT After DB insert Yes
MacroPostScriptEvaluator POST_INSERTION_SCRIPT on question After DB insert Yes
MacroPreScriptEvaluator SCRIPT on question Before insert Yes
MacroReadScriptEvaluator READ_SCRIPT on question During edit-mode load Yes

Macro vs. non-macro: Macro evaluators include the question record in the Rhino scope via putAdditionalProps().

ScriptOutputParser

Converts Rhino NativeObject / NativeArray return values to JSONObject / JSONArray for client consumption:

parseScriptOutput(Object rhinoResult)
  → NativeObject → JSONObject (recursive)
  → NativeArray  → JSONArray  (recursive)
  → primitive    → direct value

8. Producer Variable Mapping

Package: com.glideapp.servicecatalog.producer.variables

ProducerTargetVariableMapper

Purpose: In edit-mode, populates variable form fields from an existing target record.

Builder pattern:

ProducerTargetVariableMapper.builder()
    .producerId(id)
    .targetRecordId(targetId)
    .prefix("item_option_")
    .prettyName(displayName)
    .ignoreScope(false)
    .build()
    .getMap()  // returns Map<fieldName, value>

Extraction logic: 1. Load questions from CatalogItem 2. For each question, fetch value from target GlideRecord 3. Respect canRead() ACL check — skip if no read access 4. For macro/question types: run MacroReadScriptEvaluator for computed values 5. Return populated map

CompositeProducerTargetVariableMapper

Extends: ProducerTargetVariableMapper

Handles: Multi-row ProducerSets — produces JSONArray for multi-row sets, single value for ONE_TO_ONE.

ProducerTargetManagerJS

Purpose: Exposes variable mapping to server-side Rhino scripts.

// callable from catalog scripts:
ProducerTargetManagerJS.getVariableMapFromTargetRecord(producerId, targetRecordId, params)

9. OrderGuide Sequencing

Package: com.glideapp.servicecatalog.order_guide_sequencing

OrderGuideSequencing

Purpose: Integrates with Process/Activity Designer (PAD) to create a workflow process that sequences order guide items as parallel activities.

Key Methods: | Method | Purpose | |--------|---------| | shouldProcessImmediately() | Check if item is in active PAD activities (for mixed carts) | | generateSequence() | Create PAD process + trigger + activities | | generateSequenceWithRollback() | Wrap generation in RollbackContext |

PAD integration flow:

1. Check if sequencing enabled on order guide
2. Create Process Definition (PAD) record
3. Create trigger:
   condition = "approval=approved
               ^order_guide={guideId}
               ^order_guide.sequencing_processISNOTEMPTY
               ^order_guide.sequencing_process={processId}"
4. Create lane (workflow lane)
5. For each included item in guide → create Activity
6. Return process sys_id

Filtering: Only processes items with approval=approved — unapproved items wait.

OrderGuideSequencingJS

Extends: ScopedScriptableObject
Purpose: Rhino wrapper callable from catalog scripts.

var seq = new OrderGuideSequencingJS(orderGuideId);
seq.generateSequence();

OrderGuideSequenceFulfillment

Purpose: Executes an individual sequenced order guide item when PAD triggers it.

Flow:

1. Create staging cart: "staging_{orderGuideId}_{requestId}"
2. Fetch cart item from staging cart
3. Switch session user to cart owner (for RP script context)
4. ProducerValve.produceItemReturnTargetRecord()
5. Delete staging cart item
6. Return target record

10. Wizard Framework

Package: com.glideapp.servicecatalog.wizard

Architecture

SCWizard (immutable, cached)
    ├── SCWizardStep[]
    │       ├── SCWizardSection[]
    │       │       └── SCWizardQuestion[]
    │       └── SCWizardActionMapping[]
    └── producer (CatalogItem sys_id)

SCWizardInstance (runtime, wraps SCWizard)
    ├── mode: NORMAL | TEMPLATE
    ├── editMode: boolean
    ├── SCWizardStepInstance[] (computed, includes synthetic steps)
    ├── scripts: Map<ON_LOAD|ON_CHANGE|ON_SUBMIT, scripts>
    └── policies: List<CatalogUIPolicy>

WizardProcessor (request handler)
    ├── fetch() → load wizard metadata + populated data
    └── submit() → save template or delegate to ProducerValve

SCWizard

Immutable, translation-aware, cached in WIZARD_CACHE.

Key Fields: | Field | Purpose | |-------|---------| | fId | Wizard sys_id | | fName | Translated name | | fType | Wizard type | | fProducer | Associated CatalogItem (producer) sys_id | | fLayout | INLINE or TABBED | | fShowReviewStep | Whether to display review/submit step | | fSteps | Immutable List | | fCacheable | Cache eligibility flag |

Hierarchy loading:

SCWizard.createWizardRecord(gr)
  → getSteps(wizardId)
      → for each step: load sections
          → for each section: load questions
              → for each question: load action-mappings

SCWizardInstance

Runtime decorator over SCWizard. Adds: - Mode-specific step computation (TEMPLATE mode adds Details + OwnerOverride steps) - Lazy-loaded client scripts (ON_LOAD, ON_CHANGE, ON_SUBMIT) - Lazy-loaded UI policies

TEMPLATE mode synthetic steps: 1. Details step — wizard name, short_description, available_for, scope 2. Wizard steps (filtered: exclude hide_on_template steps) 3. OwnerOverride step — manage template ownership/availability 4. Review step (if showReviewStep=true)

WizardProcessor

Purpose: Handles fetch and submit requests for wizards.

Fetch flow:

fetch(requestMap)
  1. Validate wizard exists and user canView()
  2. Determine target record (new vs. existing sys_id)
  3. Determine mode (TEMPLATE vs. NORMAL)
  4. TEMPLATE mode:
     a. Load template questions
     b. Apply template values to questions
     c. Apply template UI properties (mandatory/readonly/restricted)
  5. NORMAL mode:
     a. Load target record (edit) or create temp record (new)
     b. Populate from target record (ProducerTargetVariableMapper)
     c. Apply template values/UI properties
     d. Mark target record fields readonly if user lacks write ACL
  6. Return wizard as JSON with target record data

Submit flow:

submit(requestMap)
  If TEMPLATE mode → CatalogTemplateManager.save()
  If NORMAL mode   → ProducerValve.produceItem()

Scope resolution: target record scope → parent target scope (chained wizard) → current app scope

CatalogWizardManager

Static utilities + WIZARD_CACHE: - getWizardMeta(wizardId) — retrieve/cache SCWizard (TypeSafeCacheManager, capacity 300) - getWizardInstance(wizard, templateMode, editMode) — create SCWizardInstance - applyQuestionValues(questions, valueMap) — bulk populate question values - applyQuestionTemplateValues(questions, template) — apply CatalogTemplate overrides - applyQuestionUIProperties(questions, template) — apply mandatory/readonly/restricted flags - applyMappedFieldAccess(questions, targetRecord) — mark readonly if target field not writable

Cache invalidation: On changes to SC_WIZARD_* and SC_VARIABLE_ACTION_CONFIG_WIZARD_QUESTION_MAPPING tables.

CatalogWizardHandlerJS

Extends: ScopedScriptableObject
Purpose: Exposes wizard operations to server-side Rhino scripts.

CatalogWizardHandlerJS.fetch(requestParams)
CatalogWizardHandlerJS.submit(requestParams)
CatalogWizardHandlerJS.canAccessTarget(requestParams)
CatalogWizardHandlerJS.getOwnerOverridesQuestionsAssociatedToWizard(wizardId)

11. Script Utilities

Package: com.glideapp.servicecatalog.script

CartBatcher

Extends: GlideScriptableObject (Rhino)

Purpose: Groups cart/request items by delivery frequency for batch ordering display.

// Rhino-callable:
CartBatcher.getFrequencyList(cartId)        // ["monthly", "yearly"]
CartBatcher.getFrequencyItems(cartId, freq) // item sys_ids for frequency
CartBatcher.getFrequencyLabel(freq)         // translated label
CartBatcher.getFrequencyListAndRenderCols() // frequencies + OG-grouped items

CatalogCMS

Rhino-callable checks for CMS context:

CatalogCMS.isCMSSite()                    // is request in CMS block context
CatalogCMS.canViewBreadcrumbCrossSite()   // category cross-site ACL

PageFinder

URL navigation utilities:

PageFinder.getSearchPage()  // search page URL
PageFinder.getHomePage()    // home page redirect URL

Delegates to CatalogURLGenerator.

Summary: Script Execution Flow (Record Producer)

User submits catalog item form
       ↓
ValveProcessor.dispatch(request)
       ↓
ProducerValve.before()
       ↓
  1. Create target GlideRecord (table from catalog item config)
  2. Apply CatalogTemplate (if any)
  3. Map variables from HTTP params (or target record in edit mode)
  4. Script chain:
       SaveScriptEvaluator.evaluate()
         → if save_option == SAVE_IN_DESTINATION: run SAVE_SCRIPT
       PreScriptEvaluator.evaluate()
         → run SCRIPT (pre-insertion)
       MacroSaveScriptEvaluator (for each macro question)
       ---- GlideRecord.insert() / update() ----
       PostScriptEvaluator.evaluate()
         → run POST_INSERTION_SCRIPT
       MacroPostScriptEvaluator (for each macro question)
  5. Log to sc_item_produced_record
  6. Redirect to configured URL

All Rhino scripts receive the target GlideRecord and the catalog item's script record as globals in their scope.
