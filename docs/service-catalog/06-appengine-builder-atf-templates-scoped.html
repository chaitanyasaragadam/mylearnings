1. AppEngine Template Unloading System

Package: com.glideapp.servicecatalog.appengine.template.payload.unloader

Purpose: Serializes catalog metadata for cross-scope delivery (e.g., app engine template payloads). The key challenge is that variable sys_ids change between instances; the unloader tokenizes these references so they can be re-resolved after deployment.

Tokenization Mechanism

Variable references in template records use the format IO:{32-char-hex-sys-id}. The unloader replaces these with ScanVariable tokens (symbolic names) so the payload is portable across instances.

Regex pattern used: (IO:)[0-9A-Za-z]{32}

AbstractCatalogVariableRecordUnloader

Abstract base for all record unloaders.

Key Method:

stripIOPrefix(value)
  // Finds IO: prefixed sys_ids in field values
  // Creates ScanVariable tokens for each reference
  // Returns tokenized string safe for cross-instance delivery

Concrete Unloaders

Unloader Table Field Tokenized
CatalogUIPolicyRecordUnloader catalog_ui_policy catalog_conditions
CatalogUIPolicyActionRecordUnloader catalog_ui_policy_action catalog_variable
CatalogScriptClientRecordUnloader catalog_script_client cat_variable

Payload structure:

Record data
  → IO: references detected
  → Replaced with ScanVariable tokens
  → JSON serialized for transport
  → On target instance: tokens resolved back to local sys_ids

Unloader Helpers (unloader/helpers/)

Helper classes for specific field-type tokenization, field scanning, and reference resolution strategies.

2. Catalog Builder (Question Designer)

Package: com.glideapp.servicecatalog.builder

Purpose: The Catalog Builder is the drag-and-drop UI tool for designing catalog item forms. Developers use it to add/remove/reorder questions, variable sets, and policies on a catalog item.

QuestionDesignerJS (Entry Point)

Extends: ScopedScriptableObject

Static Rhino methods:

QuestionDesignerJS.createResource(params)   // create question/container/variable_set
QuestionDesignerJS.updateResource(params)   // update question/container/variable_set
QuestionDesignerJS.deleteResource(params)   // delete question/container/variable_set
QuestionDesignerJS.updateQuestionOrders(orders) // reorder questions

Actions handled: "container", "question", "question_set"

Session context: CatalogUtil.setSessionProperties() ensures correct scope before DB operations.

QuestionDesignerReorder (LCS Algorithm)

Purpose: Minimizes the number of order-field DB updates when questions are reordered. Uses Longest Common Subsequence (LCS).

Key Data Structures:

QuestionOrderList (DoublyLinkedList)
  └── QuestionNode
        └── QueueNode

Algorithm:

computeNewQuestionOrders(existingOrder, newOrder)
  1. Run LCS to find common subsequence between old and new question order
  2. Only update order values for questions NOT in the LCS
  3. Allocate new order values in gaps between unchanged questions
  4. Rebalance gaps if insufficient space between adjacent nodes

updateQuestions(changedOrders)
  → Updates item_option_new and io_set_item (variable sets)

Result: Typically only 1-3 DB updates instead of updating all question order values.

DAO Layer

QuestionDAO

CRUD for item_option_new records:

Method Purpose

3. Automated Testing Framework (ATF)

Package: com.glide.automated_testing_framework.runner.step

ReplayRequestItemRunner

Extends: AStepRunner
Implements: ICatalogConstants

Purpose: ATF step that replays a previous catalog request item — orders the same item again for regression testing.

Step flow:

1. Read original request_item sys_id from step input
2. Create new Cart (sc_cart)
3. cart.addToCart(catalogItemId)  ← adds item to new cart
4. Copy variable values from original request_item:
   a. Load questions via VariablePoolQuestionSet
   b. For each question: get value from original → set on new cart_item
5. RequestNew.copyCart()  ← submits cart, creates new request
6. Verify new sc_req_item was created
7. Set step output:
   TABLE = "sc_req_item"
   REQUEST_ITEM = new request_item sys_id

Integration with catalog: - Uses VariablePoolQuestionSet for variable loading - Uses standard Cart / RequestNew flow — same path as live ordering - Validates catalog item still active and orderable

4. Execution Plan Engine

Package: com.glide.execution_plan

Purpose: Models and calculates delivery timelines for catalog requests. Represents tasks as a directed acyclic graph (DAG) with dependency tracking.

TaskToken (Core Node)

Implements: Comparable (ordered by path length offset)

Fields: | Field | Purpose | |-------|---------| | fDeliveryTime | Duration for this task | | fName | Task display name | | fId | Task sys_id | | fOrder | Explicit order value | | fPredecessors | Set of predecessor TaskToken sys_ids | | fSuccessors | Set of successor TaskToken sys_ids | | fPathLength | Computed GlideDuration (longest path to this task) | | fType | TYPE_NORMAL (0) or TYPE_APPROVAL (1) | | fUponReject | ON_REJECT_CANCEL (0) or ON_REJECT_GOTO (1) | | fUponRejectGoto | Target token sys_id for GOTO rejection |

Key Methods: | Method | Purpose | |--------|---------| | shouldSkip() | Evaluate Filter.checkRecord + condition_script | | getPredecessorApprovalIds() | Recursive stack traversal finding TYPE_APPROVAL ancestors with ON_REJECT_CANCEL | | getDueDate(basis, calendar) | basis + pathLength + calendar | | getExpectedStart(basis, calendar) | basis + pathLength - deliveryTime + calendar |

AbstractExecutionPlan

Static cache: PrivateCache "zzDeliveryPlanCacheyy" (capacity 100)

Static factory:

AbstractExecutionPlan.get(planId)
  1. Check cache
  2. If miss: load via loader, populate cache
  3. Return plan

Fields: | Field | Purpose | |-------|---------| | fPlanID | Delivery plan sys_id | | fCalendar | GlideCalendar for working hours | | fTaskTable | Target task table | | fStart | START sentinel token | | fTokenMap | Map |

Key Methods: | Method | Purpose | |--------|---------| | calculatePathLengths() | Recursive DFS: each token's path = max(predecessor paths) + self delivery time | | getMaxPathLength() | Maximum path = total plan duration (respects GlideCalendar working hours) | | getTokenMap() | Return all tokens | | getSortedTokenList() | Tokens sorted by Comparable (path length offset) |

ExecutionPlan

Extends: AbstractExecutionPlan

Loading strategy determined at construction: - AdvancedPlanLoader — when explicit sc_cat_item_dt_mtom relationships exist - BasicPlanLoader — when tasks are ordered by order field only

Plan Loaders

BasicPlanLoader (implicit order)

Algorithm:
1. calcSequence() — sort tasks by order field
2. calcRudder() — group tasks into "islands" by order value
   Example: orders [100, 100, 200, 300] → 3 islands
   Island 1: [100, 100] (parallel)
   Island 2: [200]
   Island 3: [300]
3. setTokens() — create sequential links between islands
   Island 1 → Island 2 → Island 3

Tasks within the same island execute in parallel. Islands execute sequentially.

AdvancedPlanLoader (explicit relationships)

1. Load all delivery tasks for plan
2. Query sc_cat_item_dt_mtom for DT_CHILD / DT_PARENT relationships
3. Build explicit predecessor/successor sets per token
4. Add START token as predecessor if task has no parents

5. Catalog Template Engine

Package: com.glideapp.servicecatalog.template

Purpose: Pre-defines variable values and UI property overrides (mandatory, readonly, restricted) for catalog items. Templates allow creating standardized request forms without modifying the base item.

CatalogTemplate (Immutable, Lombok Builder)

Table: sc_template

Fields: | Field | Purpose | |-------|---------| | fId | Template sys_id | | fName | Template name | | fShortDescription | Description | | fAvailableFor | Set user criteria sys_ids | | fUseTemplateScope | Use template's app scope | | fProducer | Associated catalog item sys_id | | fAllowEditUnlistedVariables | Whether non-template variables are editable | | fTemplateDetails | Lazy-loaded List |

Key Methods: | Method | Purpose | |--------|---------| | getTemplate(sysId) | Static factory | | getTemplate(gr) | Static factory from GlideRecord | | canUse(userId) | Evaluate UserCriteria via CSAUserCriteriaLoader; check entitlement override role | | save() | Persist template + cascade to CatalogTemplateDetail.save() | | getTemplateDetails() | Lazy load from DB |

CatalogTemplateDetail (Immutable, Lombok Builder)

Table: sc_template_detail

Fields: | Field | Purpose | |-------|---------| | fTemplate | Parent template sys_id | | fAppliesToSet | true = variable set; false = individual question | | fSet | Variable set ID (if appliesToSet=true) | | fQuestionId | Question sys_id (if appliesToSet=false) | | fValue | Pre-filled value | | fMandatory | Override mandatory flag | | fReadonly | Override readonly flag | | fVisible | Override visible flag | | fRestricted | Override restricted flag |

Constraint: readonly=true forces mandatory=false and restricted=false (you can't require user input on a readonly field).

Key Methods: | Method | Purpose | |--------|---------| | save() | Update or insert, using APPLIES_TO_SET query to find existing detail | | getTemplateDetails(templateId) | Load all details for a template | | getEntity() | Returns set or questionId per fAppliesToSet |

7. Domain Support

Package: com.glideapp.servicecatalog.domain

Purpose: Handles multi-tenant (domain-separated) catalog operations.

Key Functionality: - Domain conflict detection in carts (items must be from same domain) - Domain-aware cache keys (cache entries are domain-scoped) - Domain-specific user criteria evaluation - Cross-domain item visibility rules

Integration point: CatalogUtil.getDomain() + DomainSupport.isMultiDomain() used throughout.

9. Catalog Item Versioning

Package: com.glideapp.servicecatalog.catalog_item.versioning

CatalogItemPublishOperation

Purpose: Manages publishing workflow for catalog items. Handles draft → active version transitions.

Key Operations: - Create new version of a catalog item - Promote draft to published state - Rollback to previous version - Compare versions

Test coverage: CatalogItemPublishOperationTest validates versioning and publishing logic.

10. Cart Validators

Package: com.glideapp.servicecatalog.cart.validators

Purpose: Pluggable validation rules run before cart checkout.

Validators include: - User criteria check: Can user still order each item? - Domain check: Are all items in the same domain? - Requested-for validation: Is the requested_for user set and valid? - Quantity validation: Are quantities within allowed limits? - Active check: Are all items still active?

Pattern: Each validator implements a common interface; Cart.validateCartItems() runs all validators and aggregates errors.
