Cache, DB Listeners & Metrics

glide-servicecatalog TypeSafeCacheManagerSCStaleCacheListenerMetricsUI Policy

1. Caching Architecture Overview

The caching system is multi-layered and domain/translation-aware. Every cached object can be invalidated by DB change events, scope changes, or translation updates.

ServiceCatalogCacheUtils (utilities + helpers)
    ↓ uses
TypeSafeCacheManager<K, V>  (per-entity cache)
    ↓ stores
IServiceCatalogCacheableItem (marker interface)
    └── ServiceCatalogCacheableItemAdapter (base adapter)
        └── ServiceCatalogI18NCacheableItemAdapter (translation-aware)

4. Translation Engine

DefaultTranslationProvider (450+ lines)

Purpose: Core translation engine. Resolves translated text from multiple sources with locale fallback.

Translation sources (in priority order): 1. sys_translated_text — direct field translations 2. sys_translated — translated field records 3. Source table fields — base language values 4. sys_choice — choice label translations

getChoiceTranslation() locale fallback chain:

1. Requested locale (e.g., "fr")
2. Fallback language (e.g., "en")
3. Default system language

Key internal structures: - Synchronized maps for thread-safe table hierarchy + translated elements - getFieldTranslationsFromSourceTable() — with domain handling - getFieldTranslationsFromSysTranslated() — for Translated Field type

5. ServiceCatalogCacheUtils

Central cache utilities class.

Key Methods

Method Purpose
buildStaleCacheItemList(context) Evaluate and collect stale cache entries
cachedString(key, supplier) String caching with optimization toggle
createSysId2ObjectMap() Memory-optimized sys_id → object map
createSysIdCopyOnWriteArrayList() Thread-safe list for sys_ids
makeTranslatableObjectCacheKey(sysId, locale) Build cache key (V1: language-aware; V2: sys_id only)
transactionCacheItem(key, supplier) Soft per-transaction cache

Cache Key Strategy

Mode Key Format
V1 (legacy) {sysId}_{language}
V2 (default, isCacheApiV2Enabled()=true) {sysId} only

V2 is simpler and avoids per-language cache misses. Translation is resolved at read time rather than baked into the key.

6. DB Listeners & Cache Invalidation

CatalogDBListener

Purpose: Central dispatcher for all catalog-related DB change events. Uses Supplier pattern for lazy listener initialization.

Internal routing maps:

fCatalogInstanceListener:  table → List<BiConsumer<IDBAction, Boolean>>
fCatalogHierarchyListener: table → List<BiConsumer<IDBAction, Boolean>>

Registered listeners:

Table Listener
item_option_new VariableListenerConfigMeta
item_option_new_set VariableSetListenerConfigMeta
request_filter RequestFilterListener
catalog_script_client DBUpdateEventGenerator
catalog_ui_policy DBUpdateEventGenerator
catalog_ui_policy_action DBUpdateEventGenerator
question_choice DBUpdateEventGenerator

Event types detected: DBInsert, DBUpdate, DBDelete

ICatalogDBListenerMeta

Interface providing:

List<BiConsumer<IDBAction, Boolean>> getListenerConfig()

AbstractContextAwareCatalogListener

Abstract base with lazy listener config initialization. Subclasses provide the specific listener logic.

SCStaleCacheListener

40+ table mappings → StaleCacheFinder implementations.

Key finders:

Finder Tables Covered
StaleItemCacheFinder sc_cat_item, catalog item hierarchy
StaleCategoryCacheFinder sc_category, M2M relationships
StaleVariableSetCacheFinder item_option_new_set
StaleCatalogM2MItemCacheFinder sc_cat_item_catalog_mtom
StaleDynamicItemCacheFinder Dynamic content items
StaleUserCriteriaCacheFinder User criteria changes

FinderConfig lookup algorithm:

1. Direct lookup in fFinderConfigMap
   → found: return finder
2. Check exclusion map
   → excluded: return null (no invalidation needed)
3. Hierarchy search (table inheritance chain)
   → found: add to direct map, return finder
4. Not found: add to exclusions map

VariableSetListener

Purpose: Handles item_option_new_set DB changes.

On insert/update: - Sets default title - Sets internal_name via DBColumnNamer - Sets type field

On delete: - Cleans up references in catalog template records - Filters out deleted variable set from JSON fields: serializedData, variablesLayout, variables

13. Virtual Agent Support

Package: com.glide.catalog.virtual_agent

CatalogConversationHelperJS (1412 lines)

Purpose: Enables Virtual Agent (VA) conversations to interact with catalog items.

Rhino-callable methods: | Method | Purpose | |--------|---------| | getItem(itemId) | Full item structure with variables, data lookups, UI policies | | setValue(itemId, varName, value) | Handle variable change with policy evaluation | | submit(itemId, params) | Submit form with validation | | runChangePolicies(itemId, changedVar, params) | Execute on-change policy evaluation | | getNonConversationalReasons(itemId) | Analyze why item may not support full VA conversation |

getNonConversationalReasons() checks for: - Pricing implications - MRVS (multi-row variable sets) - Advanced reference qualifiers - Reference record limits exceeded - Client scripts present - Scripted UI policies

ConversationObject

Data holder with static fMethodMap binding field names to setter/getter pairs via reflection.

Key fields: fSysId, fName, fDescription, fVariables, fPoliciesConditionResult, fDataLookupSetters, fDynamicValueVariableConfig, fCatalogs, fFormMessages, fRecurringFrequencyDisplay

Serialization: parseResponse() / buildResponse() use reflection-based dynamic serialization via fMethodMap.

14. Cloning

Package: com.glide.catalog.cloner

CatalogCloneRequest

Extends: HierarchicalProgressWorker

Purpose: Deep-copy an existing request for one or more target users.

cloneRequest() flow:

For each userId in fUserIds:
  1. request.cloneForUser(userId)
     a. Cart.backwards() — reverse-engineer request back to cart
     b. Cart.checkoutCart() — re-submit cart for new user
  2. Track progress via ExecutionTracker
  3. Clear cart after each clone
  4. Return List<Request> of successful clones
  5. Return result URL via CatalogURLGenerator.getRequestListUrl()

Error handling: CartException failures tracked per user; successful clones continue.
